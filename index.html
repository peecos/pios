PIOS 2.0, Core, and Cotton

A unified master document for the Personal Information Operating System framework, the Cotton Context Blueprint, the Core implementation, and the agent-operated personal information architecture.

Generated: 2026-06-08 Updated: 2026-07-02 Source: PIOS 2.0 working discussions, Cotton, AWS, History, OKF comparison notes, LLM Wiki assessment, and related architecture reviews Status: Working master draft Theme: Light

1. Purpose

PIOS 2.0 defines a personal information architecture where all information about a person can be captured, preserved, organized, interpreted, retrieved, and moved without becoming dependent on any one application, platform, database, cloud provider, or agent runtime.

Foundational statement: PIOS is the open framework. Core is the usable implementation. A person's Core is their durable personal information foundation: not a brain, not a memory app, not a social profile, but the place where everything about them can be preserved and made usable by them and their trusted agents.

The architecture is built around a single information core. All incoming files, messages, app records, documents, notes, conversations, events, sensor readings, agent outputs, and operational definitions enter as raw preserved items first. Their information value is then extracted into events, metadata, glossary terms, context, knowledge notes, and derived projections.

The system intentionally separates three things that many products collapse together:

  • What happened: canonical events and immutable originals.
  • What it means: knowledge objects, context, glossary terms, interpretations, summaries, and patterns.
  • How it is used: retrieval APIs, agents, applications, dashboards, Circle conversations, and product surfaces.

This document merges the original PIOS discussions, included architecture notes, later cross-review findings, and the Cotton 2.0 Context Blueprint discussion into one coherent master documentation version. It keeps the general PIOS/Core architecture, Cotton information organization layer, AWS-native Core Managed implementation, and portable Core Self-Hosted implementation in the same master document while separating them into clearly labeled segments.

2.1 Linux and Ubuntu Analogy

The intended analogy is useful but should be kept precise:

PIOS = Linux-like open framework and specification Core = Ubuntu-like practical implementation Core Managed = hosted Core service Core Self-Hosted = portable Core VM/server distribution

PIOS should not depend on AWS, Ubuntu, Garage, DuckDB, PostgreSQL, or any other specific technology. PIOS defines what a compliant personal information core should do. Cotton defines the conceptual information organization used inside that core. Core implements both in a concrete, usable stack.

2.2 Why "Core"

The product name Core is intentionally foundational rather than cognitive or emotional. It is not the heart, mind, brain, or memory. It is the center that everything else can rely on.

  • Physical meaning: core muscles hold the body together and enable stable movement.
  • Identity meaning: core values, core beliefs, and "at my core" point to the essential self.
  • Structural meaning: the core is the center that applications, agents, relationships, and workflows can build around.

2.3 Core, Core Self, and Circle

The human model is:

Core Self private source of truth for the person and their trusted agents Inner Circle closest people and trusted AI agents Extended Circle collaborators, colleagues, professional network, limited sharing Outer Circle public or broadly shared information

Privacy is structural. Core Self is private by default. Inner Circle is trusted private sharing. Extended Circle is limited sharing. Outer Circle is public or intentionally exposed. AI agents can be treated as members of the owner's inner circle when they have delegated trusted access to Core Self.

Agent trust scope and agent profile home are separate concepts. A trusted agent may operate with Inner Circle access, while its profile, permissions, runtime files, and governance records still belong to My System.

3. Governing Principles

Principle 1
One information core. The person's information should have one canonical foundation even when applications, devices, agents, and sources remain distributed.
Principle 2
Capture first, organize later. Information should enter quickly and safely before perfect classification is attempted.
Principle 3
Originals are preserved. Raw files and source records are kept as received and are policy-immutable: protected through append-only handling, versioning, retention, governance controls, and corrective events rather than silent mutation.
Principle 4
The event log is the canonical spine. Activity, updates, captures, corrections, derived summaries, and system changes are represented as events in the append-oriented event log.
Principle 5
Storage follows information nature. Static facts, rarely changing profile items, documents, logs, streams, and high-volume sensor data should not be forced into one physical format.
Principle 6
Markdown with YAML front matter is the default semantic layer. It remains human-readable, version-friendly, portable, and easy for agents to process.
Principle 7
Derived data is rebuildable. Summaries, embeddings, graph exports, thumbnails, Parquet projections, and indexes must not become the only copy of truth.
Principle 8
Use common language first. Life domains, roles, modes, relationship terms, and organizing concepts should use terminology familiar to humans and language models wherever possible.
Principle 9
Personal nuance is captured through the glossary. Cultural concepts, nicknames, company-specific terms, product names, and personal meanings are first-class glossary items.
Principle 10
Introduce specialized systems only when needed. Databases, queues, graph stores, vector stores, search engines, workflow systems, and orchestration platforms are added only when actual data volume or retrieval needs justify them.
Principle 11
Cotton guides meaning before technology chooses format. Agents should first understand what kind of information an item represents, where it fits in the person's life, and which terms should describe it. Only then should they choose the storage format, folder, event type, metadata structure, or processing route.
Principle 12
Everything important is part of the information system. Goals, plans, projects, tasks, routines, agent profiles, agent skills, operational history, schedules, and completion records are not external administration. They are first-class information governed by PIOS and Cotton.
Principle 13
The event log is complete; owner-facing History is curated. Communications, captures, system activity, processing steps, corrections, and completed work can all enter the canonical event log. The readable History surface is derived from selected events, structured updates, manifests, summaries, and retained artifacts.
Principle 14
Classify without siloing. Core should preserve one connected personal information foundation. Separation, duplication, and audience-specific views are used intentionally for sharing, exposure control, privacy, and comprehension in Circle, not to fragment truth inside Core.
Principle 15
Retrieve summary-first. Good retrieval does not start by reading everything. Agents should choose the right navigation layer first: index, summary, History period, concept page, person page, project page, graph neighborhood, event record, or original source depending on the question.

4. Cotton - Context Blueprint

Cotton is the Context Blueprint for PIOS 2.0: the non-technical organizing canon that tells humans and agents how personal information should be understood, named, grouped, related, and applied.

Core distinction: PIOS defines the open personal information architecture. Core implements it. Cotton defines the information organization logic inside it: the principles, categories, metadata conventions, terminology, examples, and decision rules agents use when information is dropped into the system.

4.1 What Cotton Represents

Cotton is not a database schema, folder tree, ontology, or app-specific taxonomy. It is the blueprint that sits above those implementation choices and answers questions such as:

  • What type of information is this?
  • Is it about the person, the person's world, or the operation of the system itself?
  • Which life domain, role, mode, relationship, location, project, or time frame does it belong to?
  • Which title, category, tag, summary, metadata, event type, and knowledge projection should an agent use?
  • Which external standard or common vocabulary should be mapped where relevant?

The name Cotton carries the intended analogy: simple and familiar at a distance, but when inspected closely, made of many interconnected fibers. It also evokes a small cloud: a personal cloud that looks soft and simple from the outside while containing a dense network of personal context inside. Personal information behaves the same way. It looks like files, notes, events, messages, relationships, and tasks on the surface, but underneath it is a soft, layered, contextual, and evolving mesh.

4.2 Core Organizing Scope

Cotton starts with two primary personal information areas: Myself and My World. Because agents, schedules, dashboards, runtime files, and operational machinery are also information, Cotton also treats My System as a derived first-class operational area.

These areas are organizing lenses over one connected Core, not isolated containers. The same item may involve the person, their world, and their system at the same time. Agents should preserve those relationships instead of copying the item into disconnected silos.

Profile-like information must preserve this distinction. The owner profile belongs to Core Self and the Myself area. Other human profiles belong to My World as relationship/contact profiles scoped through Inner Circle, Extended Circle, or Outer Circle. Agent profiles belong to My System as delegated system actors with permissions and governance; they are not possessions, even when their runtime files, accounts, or tool credentials are managed as system assets.

Area Role in Cotton Meaning Examples
Myself Primary personal area Information about the individual as a person and owner of Core Self. Owner profile, identity, background, preferences, health, skills, values, roles, history, habits, goals.
My World Primary personal area Information about what is part of the person's life and domain. Human/contact profiles, organizations, projects, possessions, assets, places, communications, documents, responsibilities, experiences.
My System Derived operational area Information about the tools, agents, rules, interfaces, and operational machinery that maintain the person's Core. Agent profiles, skills, permissions, processing manifests, data sources, connectors, schedules, dashboards, implementation settings.

Operational flow objects should be classified by what they are about, not by the fact that software tracks them. A goal, project, task, Work Starter, or decision belongs to Myself or My World when it describes the owner's life, work, relationships, assets, or commitments. My System is for the machinery that operates Core: agent definitions, processing manifests, connector state, execution jobs, queues, runtime configuration, sync status, and system governance.

4.3 Information Stability Layers

Cotton separates information by how stable it is, because storage, review cadence, agent confidence, and retrieval behavior differ by stability.

Layer Description Examples Agent handling
Stable Information that rarely or never changes. Date of birth, origin facts, enduring principles, long-term identity statements. Store with strong provenance; change only through explicit correction or versioned update.
Slowly changing Profile-like information that evolves but should not churn daily. Skills, roles, preferences, health baselines, relationships, responsibilities. Maintain as living profile notes with last-reviewed dates and source-linked updates.
Dynamic Operational information that changes frequently. Tasks, projects, plans, events, messages, calendar items, work sessions, updates. Represent as events, task/project records, logs, status views, and derived summaries.

4.4 Personal Operating Context

Cotton must also describe how a person operates, not only what information exists about them. This layer is the person's Personal Operating Context: the reusable context that helps agents understand goals, responsibilities, motivation, working modes, time windows, review rhythm, and AI support preferences.

Personal Operating Context is generic in the PIOS framework. Each person fills it differently, but the information types are common enough to be first-class Cotton concepts.

Operating concept Meaning Example for John Doe
Motivation foundation What gives energy, meaning, satisfaction, or resistance. John is motivated by useful product work but tends to avoid repetitive admin.
Responsibility areas What the person must decide, maintain, deliver, or protect. Client delivery, family responsibilities, business development, and personal health.
Revenue and sustainability model How current work funds continuity and what future revenue model is desired. Consulting funds the transition toward productized services.
Leverage engine Tools, assets, systems, templates, workflows, and products that make future work easier or more scalable. Reusable proposal library, AI-assisted delivery workflow, and onboarding templates.
Working modes The different kinds of working a work item may require and the conditions where they fit. Deep work, client delivery, admin, learning, voice work, travel work, and review.
Base Week A realistic planning grammar for how the person's week usually flows. Mornings are better for orientation; 90-minute afternoon windows fit writing; small pockets fit admin.
Time windows Available work opportunities described by duration, energy, location, tools, and context. 15-minute pocket, 90-minute focus window, walking voice window, travel window.
Attention signals Why something may need attention now, without turning priority into a static label. Responsibility, sustainability, leverage, alignment, timing, commitments, window fit, and mental-load reduction.
AI support rules What agents may observe, summarize, prepare, recommend, draft, remind, or execute with approval. AI drafts a first outline, but John approves the final client-facing promise.

Design rule: the owner experience should not expose generic productivity rules as commands. Cotton should store the person's operating foundation and let agents use it underneath so the system feels like it understands how the person works.

4.5 Cotton Guiding Principles

Cotton is deliberately principle-led. It should not try to predefine every future folder, tag, field, or title. Instead, it should give agents enough principles and examples that new cases can be classified consistently even when the exact case was never documented.

Principle Guidance for agents
Principles before structure When the exact category or field is missing, reason from Cotton principles first. Do not invent a new permanent taxonomy value until the gap repeats or the owner approves it.
No internal silos Do not split the owner's Core into isolated life silos. Classify items across domains, roles, people, projects, sources, context, events, and axes while preserving one connected information foundation.
Migrate by functional boundary When an owner already has an existing information system, transition by functional system area, not by folder, app, or export size. History, dashboards, communication, work objects, profiles, agents, raw traces, takeouts, and repositories may cross many source folders but should be mapped by what role they play in the owner's system.
Separation is for exposure control Create separated views, copies, summaries, or shared packets when information leaves Core Self for Inner Circle, Extended Circle, Outer Circle, dashboards, exports, or public surfaces. The separation exists to control audience, context, and privacy, not to create a second truth.
Personal meaning plus common language Use ordinary natural-language labels first so humans and language models understand the system. Capture personal nuance, nicknames, cultural concepts, and local meanings in the glossary.
Standards as mappings, not masters Map to schema.org, calendar/task formats, contact/identity formats, geography standards, health standards, finance standards, or other domain vocabularies where useful. Do not force personal meaning to obey an external standard.
Domains are dimensions, not destinations Life domains support broad filtering and reasoning. An item may belong to multiple domains or none; more precise meaning often comes from role, mode, person, project, source, event, or purpose metadata.
Classify by information nature Separate information area, type family, stability layer, source type, origin class, technical form, and life domain. Do not make one axis carry the meaning of another.
Stability determines handling Stable information needs strong provenance and careful correction. Slowly changing information needs profile-style review. Dynamic information needs event links, status updates, and current projections.
Source bias must be recorded Do not infer a person's full life architecture from one corpus. A work-heavy archive is strong evidence for work, system, and execution patterns but weak evidence for health, family, recreation, possessions, lived finance, and stable identity.
Originals stay intact; projections stay useful Preserve raw originals and source references. Create Markdown/YAML notes, events, summaries, indexes, HTML detail pages, and knowledge views as projections that can be corrected or rebuilt.
Everything operational is information Goals, plans, tasks, routines, Work Starters, updates, dashboards, schedules, agent profiles, skills, permissions, and processing manifests are part of Cotton. They are not outside administration.
Examples guide, but do not freeze Use examples as reference patterns. If a new item does not fit, preserve it, classify what is known, mark uncertainty, and propose the smallest extension that keeps the model understandable.
Technical implementation follows conceptual organization Choose folders, object-store buckets, databases, indexes, graphs, vectors, queues, and processing systems after the conceptual meaning is clear. Cotton should remain portable across AWS, self-hosted, local, and future implementations.

Agent rule: when classification is uncertain, preserve the original, attach the best available metadata, explain the uncertainty, and ask whether the missing concept should become a glossary term, taxonomy value, life-domain extension, context value, or implementation detail.

4.6 Building Cotton

The first formal Cotton version should be built in four passes:

  1. Define Cotton principles: information ownership concepts, categorization rules, relationship rules, abstraction levels, naming conventions, and standards preferences.
  2. Analyze historical data: mine PIOS work, NotebookLM exports, ChatGPT history, existing vaults, and personal archives for two outputs: person-specific glossary/domain terminology and a general map of information types that appear in an individual's life.
  3. Build the framework: produce categories, glossary seeds, taxonomy values, metadata examples, domain relationships, agent-readable instructions, and human-readable reference pages.
  4. Test with real data: have agents organize actual information, then review what worked, what was ambiguous, and which missing concepts need to be added.

4.7 Historical Mining Outputs

The historical mining pass for Cotton should produce two separate outputs. They must not be blended, because they serve different layers of the system.

Output Purpose How it feeds Cotton
General information-type map Identify the recurring kinds of information an individual's life contains. Feeds type families, reference templates, metadata guidance, source routing, and agent classification behavior.
Personal glossary and terminology seeds Identify person-specific names, concepts, organizations, projects, products, terms, and meanings. Feeds the personal glossary after owner review; it should not automatically become the generic PIOS/Cotton canon.

The first mining pass over historical ChatGPT material produced an important coverage caveat: the corpus was heavily work-oriented. That makes it strong evidence for Career/Work, My System, agent operations, product strategy, execution flow, glossary seeds, and knowledge-work outputs, but weak evidence for everyday personal domains such as Health, Family, Recreation, Possessions, and lived personal Finance. Source bias should be recorded as part of the mining result, not silently averaged away.

Coverage validation should test the framework, not require personal-data completeness. Available owner sources can be used where convenient, but Cotton should also be checked against public examples, synthetic profiles, mock records, and common export schemas so the model stays general for any person's Core.

4.8 Transition-Time Disposition

Existing-system transitions need one additional Cotton axis: core_disposition. It records how a source, artifact, folder, app, or record should relate to Core during transition. This is different from sensitivity, source type, origin class, storage zone, or knowledge object type.

Disposition Meaning
canonical Core should retain the source or object as part of the owner's durable canonical information foundation.
derived_projection The item is useful as a view, summary, interface, index, detail page, manifest, or analysis output, but it can be regenerated or corrected from retained sources and events.
external_referenced Truth remains outside Core, such as in Git, a live app, an external provider, or a machine-local runtime. Core records a reference, source rule, or event trail without copying the full source.
staged The item is temporarily present for intake, screening, transformation, or review, but has not been promoted into retained Core state.
excluded The item is deliberately not imported. The exclusion reason should be recorded so future agents do not repeatedly rediscover the same boundary.
guarded_pending The item may belong in Core later, but sensitivity, privacy, legal, retention, deletion, or owner-approval questions must be resolved first.

core_disposition is especially useful when a folder is mixed: one file may be canonical, another may be a derived projection, another may stay externally referenced, and another may be excluded. Agents should record disposition per item where possible rather than approving whole-folder migration.

4.9 Seed Information-Type Families

The mining pass found six stable information-type families. These are not replacements for the five Core data zones. They are Cotton-level type families that help agents understand what kind of meaning an item has before selecting storage, metadata, projections, or retrieval routes.

Family Typical area Typical stability Examples
Identity and Self myself stable / slowly_changing Identity facts, profile dimensions, values, preferences, skills, professional bio, personal glossary.
People and Relationships my_world slowly_changing / dynamic Person profile, contact profile, collaborator role, organization profile, relationship or interaction record.
Knowledge and Work Outputs my_world dynamic to slowly_changing Strategy document, research report, article draft, presentation, framework, glossary, design asset, transcript.
Functional Flow and Execution myself / my_world dynamic Goal, target, plan, project, task, routine, decision, open thread, status snapshot, result, update, dashboard.
System Artifacts my_system slowly_changing / dynamic Agent profile, skill definition, automation configuration, tool setting, schema, architecture spec, governance rule.
Assets and Records my_world mixed Domain name, device, possession inventory item, invoice, account, credential, health metric, travel plan, recipe, family record.

Functional flow records belong to myself or my_world when they describe the owner's goals, plans, projects, tasks, decisions, routines, commitments, or results. They belong to my_system only when the flow object is itself Core operational machinery, such as connector repair work, agent job orchestration, processing manifests, maintenance queues, or deployment tasks.

These type families should remain extensible. They provide a compass for unseen cases, not a closed ontology. Agents should classify the type, preserve the reason, and propose additions when a new type appears repeatedly.

5. Architecture Overview

PIOS is best understood as a layered architecture around a durable object-storage core and an event-first information model.

External sources, direct uploads, apps, agents, sensors, conversations | v Universal Inbox | v Organizer / Inbox Agent | +-- preserve raw item ---------------------> Originals | +-- create minimal event ------------------> Events | +-- extract metadata and context ----------> Events / Knowledge / System | +-- update glossary if needed -------------> System Glossary | +-- create or update living notes ---------> Knowledge | +-- build optional projections ------------> Derived | v Retrieval routing for agents, APIs, apps, Circle, and owner workflows

5.1 Five Core Data Zones

Zone Purpose Mutability
Originals Everything exactly as received: files, exports, emails, audio, PDFs, images, source records, data takeouts. Policy-immutable
Events Append-only canonical personal event log describing what happened and when it was recorded. Append-only
Knowledge Living Markdown projections: people, organizations, projects, roles, locations, decisions, habits, topics, experiences. Mutable but traceable
Derived Rebuildable outputs: extracted text, summaries, aggregates, Parquet files, embeddings, graph exports, thumbnails. Rebuildable
System Definitions that operate and interpret the core: glossary, schemas, connectors, sync state, manifests, agent definitions, configuration. Controlled/versioned

The System zone is more load-bearing during transition than a normal configuration folder. It holds the source definitions, disposition records, processing manifests, migration manifests, event registries, schema versions, review checklists, approval records, source rules, connector state, agent definitions, and parity/decommission evidence that let Core explain how information moved from an old system into the new one. Agent/runtime records should live as system artifacts unless they are deliberately promoted into knowledge as stable concepts, policies, or owner-facing explanations.

5.2 Conceptual Model Versus Physical Storage

The conceptual architecture and physical storage format do not need to be identical.

Conceptually Physically
One logical personal event log Partitioned JSON files and NDJSON batches by source and time
One logical event per row One JSON object for ordinary events; immutable batch files for high-volume events
Living knowledge about the person and world Markdown files with YAML front matter
Fast analysis Derived Parquet partitions queried with DuckDB or equivalent
Semantic and graph retrieval Embeddings, vector indexes, graph exports, or graph databases only as justified

Cotton metadata may be rendered into tables, catalogs, indexes, graph exports, vector indexes, or databases for retrieval and operational efficiency. Those structures are projections unless explicitly promoted by governance. The canonical retained semantic representation remains source-linked originals, event-log records, and Markdown/YAML knowledge objects.

6. Information Model

PIOS treats information according to its nature, not according to one predefined schema. A birthday, a legal document, a voice note, a chat transcript, a meeting, a heart-rate stream, and an agent definition have different storage and retrieval needs.

6.1 Storage Decision Classes

Information Type Example Primary Form Notes
Permanent or rarely changing fact Date of birth, legal name, eye color Markdown knowledge note with front matter Updates should preserve history through events or explicit versions.
Document or file PDF, contract, photo, data export Policy-immutable original plus extracted metadata Text extraction and summaries are derived.
Message or communication Email, chat, DM, agent conversation Original source record plus event Participants and context become retrieval dimensions.
Activity event Meeting started, walk completed, document created JSON event Can point back to originals and forward to knowledge objects.
High-volume stream Heart rate, GPS, browser activity Policy-immutable NDJSON batches Parquet projections can be generated for analysis.
Operational definition AGENTS.md, skill definition, role assignment System definition in Core In-use copies are synced from Core to execution environments.

6.2 Metadata Flexibility

PIOS and Cotton metadata is as-needed, extensible, and reprocessable unless a field is explicitly required by a minimal event envelope, source contract, workflow state machine, compliance rule, or implementation-specific interface. Agents should add metadata when it improves preservation, retrieval, interpretation, sharing control, automation, provenance, or future reasoning. Missing metadata should not be treated as a negative fact.

Most attributes can be added later. If a new dimension becomes useful, agents may add fields to selected documents, events, profiles, Work Starters, source definitions, or derived views, then backfill or reprocess a source, project, time period, domain, or profile family as needed. This keeps Core flexible: capture is not blocked by incomplete metadata, and later organization does not require mutating canonical originals.

Required fields should stay rare and justified. The stable core should be limited to enough identity, time, type, source, status, and provenance metadata to preserve traceability and workflow reliability. Richer Cotton attributes remain optional until real use proves they should become part of a specific object template or projection.

6.3 Default Semantic Container

Markdown with YAML front matter is the default for living semantic knowledge. It should be used until the nature or volume of the data clearly requires something else.

Primary Core records should distinguish original source time from Core record time. source_original_time records when the source item originally happened, was created, or became true. primary_created_time records when Core promoted the source into a primary semantic record. primary_updated_time records later Core-side edits, enrichments, or corrections. Local templates may still expose created_at and updated_at, but those fields should be interpreted as Core record timestamps unless explicitly source-qualified.

6.4 Bundle and Markdown Conventions

PIOS can borrow a small set of practical bundle conventions from open Markdown knowledge formats without depending on any external standard. The useful idea is that a meaningful set of knowledge can be packaged as a self-contained directory tree of Markdown files with YAML front matter, links, indexes, and optional generated views.

Convention PIOS Use Rule
Core Bundle An exportable or distributable set of Core information. It may be scoped to a subset, such as an owner profile bundle, person bundle, project bundle, organization bundle, travel bundle, or public sharing bundle; or it may be a full Core Export Bundle used for migration, rebuild, or takeover. A Core Bundle should be reconstructable at its chosen scope: it should carry its files, front matter, links, manifests, provenance, and enough instructions to import, inspect, or rebuild relevant projections. When the bundle contains the whole Core, call it a full Core Export Bundle to distinguish it from scoped sharing or testing bundles.
Cotton Bundle A Core Bundle focused on Cotton organization: glossary terms, taxonomy values, templates, principles, context examples, life-domain examples, or a structured subset such as a project or person context blueprint. Cotton Bundles should preserve meaning and classification logic, not only files. They are useful for sharing, migration, testing, and agent onboarding.
index.md Optional navigation file at a bundle root or directory level. Use for progressive disclosure so a human or agent can inspect what exists before loading every file. It may be generated from front matter and directory contents.
log.md Optional human-readable update projection for a bundle or directory. Never treat log.md as canonical truth. The canonical change record remains the event_log, source manifests, processing manifests, commits, retained artifacts, and governed update records.
Minimum concept document A Markdown file with parseable YAML front matter and a non-empty type. Everything else should remain optional unless a workflow, event envelope, source connector, export contract, or sharing rule justifies making it required.
Permissive consumption Agents, importers, renderers, and retrieval systems should tolerate incomplete or evolving bundles. Do not reject a bundle only because optional fields are missing, a type is unknown, extra keys exist, links are broken, or index.md is absent. Preserve unknown keys when round-tripping.

This convention reinforces the PIOS rule that capture should not wait for perfect structure. Required fields stay rare; richer Cotton metadata can be added, backfilled, or reprocessed later. Bundles are distribution and navigation units, not a replacement for Core's source-of-truth model.

Transferable Core artifacts: a reproducible Core Managed deployment should be understood as three versioned artifacts, not one opaque image. The Core Template is the parameterized infrastructure shape. The full Core Export Bundle is the owner's complete exportable data payload. The Provisioning Manifest binds a specific template version, owner, account, region, security posture, and optional bundle hydration into one rebuildable Core instance. Structure is golden and repeatable; data is owner payload; the manifest binds them.

Compatibility contract: deployment independence requires a stable external Core API, provider-independent canonical references such as core://events/<event_id>, explicit Core/API/export-format versions, capability discovery, and a validated export/import path. Provider object paths, cloud identifiers, indexes, and queues belong in implementation adapters or provenance, not in canonical meaning. The normative requirements and implementation goals are published in the Core Distribution and Compatibility Specification v0.1 and its roadmap.

7. Inbound Flow

All information enters through a universal inbox, regardless of source. The source can be a file upload, API call, connected account, recurring stream, voice note, data takeout, email, calendar feed, health sensor, agent output, or manual note.

7.1 Universal Inbox Rule

The inbox is the only normal path into Core. The agent responsible for the inbox performs triage, preservation, initial event creation, and routing. Information can be processed at different speeds:

  • Immediate triage: identify source, type, time, original reference, and basic event.
  • Background enrichment: extract text, metadata, glossary terms, context, people, topics, and links.
  • Periodic interpretation: build patterns, habits, aggregates, summaries, graph exports, and retrieval indexes.

7.2 Core Service Interface

Core should be usable as a service without requiring the owner or a connected application to design schemas first. The external interface should expose a few simple capabilities while keeping the internal information architecture consistent.

Interface Purpose Core Behavior
Drop/upload Submit one-off files, notes, exports, recordings, or documents. Preserve original, create event, triage format, enrich later.
Connected source registration Connect an external source such as email, calendar, health, banking, cloud drive, or chat. Create source definition, sync state, event mapping, deletion/change behavior, and processing policy.
Stream ingestion Accept ongoing high-volume or recurring data from a source. Process according to source-level rules, batch where needed, and avoid treating every item as a standalone document.
Retrieval API Allow agents, apps, and owner surfaces to ask for information. Route across keyword, structured, semantic, graph, or hybrid retrieval and return provenance.
Update submission Let agents and apps submit learnings, corrections, decisions, or new outputs. Send updates back through the inbox path so the same triage, provenance, and event logic applies.

7.3 Connected Sources

A connected source is not the long-term owner of the person's accumulated record. It is an input channel. The external service may remain the original source of truth for its own current records, but Core becomes the aggregated source of truth for the person's cross-source record.

When source data is promoted into primary Core records, the original source item should remain preserved where possible. Promotion does not erase source identity or source timestamps. A primary Core record has its own Core-created timestamp, while the source item keeps the original time, creation time, update time, and provenance supplied by the originating system.

Concept Meaning
Original source of truth The external system where an item originated: Gmail, Google Calendar, Apple Health, bank, file system, chat service, or app.
Aggregated source of truth Core's preserved, normalized, event-based, cross-source personal record.
Connector state Sync cursors, backfill status, deletion markers, last processed IDs, and source configuration.
Tombstone An event or state marker indicating that a source item was deleted or became unavailable without erasing historical context.

7.4 Source Promotion Lifecycle

Imported archives, live connectors, and one-off source drops should move through an explicit lifecycle. This prevents Core from bulk-promoting raw material into History, knowledge, or owner-facing summaries before its value, risk, and mapping are understood.

Stage Meaning Promotion rule
Parked Source file, export, upload, or connection exists and is preserved, but no substantive processing has happened. Record source existence and provenance only.
Analyzed Core understands what the source contains, its risks, useful data types, likely value, and processing plan. Write an analysis note or source definition before extraction begins.
Mapped Basic structured extraction exists: stable IDs, timestamps, source records, entities, and candidate events. Create primary records only where source mapping is clear and provenance is preserved.
Enriched Selected high-value subsets receive deeper extraction, glossary mapping, context, people, projects, topics, summaries, or embeddings. Enrich deliberately; do not enrich everything by default.
History-included Selected records are promoted into owner-facing History, summaries, or completed-work narratives. Always staged and explicit. Never bulk-promote a whole source into owner-facing History merely because it was imported.

7.5 Source-Level Processing

The agent should not always treat every item as isolated. For recurring sources, the source itself needs a processing definition:

  • What kinds of data can this source produce?
  • Which fields are stable identifiers?
  • What is the update, deletion, or correction model?
  • Which events should be created?
  • Which data should be batched?
  • Which knowledge objects may be updated from this source?
  • Which conflicts can occur with other sources?

Source-aware handling should distinguish basic mapping from deeper enrichment. Basic mapping may run automatically when it only preserves source identity, timestamps, stable IDs, raw references, and obvious baseline records. Deeper enrichment should require a standing owner preference, an explicit owner request, or an approved action tag when it infers meaning, detects relationships, proposes profile implications, generates summaries, performs costly analysis, or changes what enters retrieval, summaries, or owner-facing History.

Import concept Meaning Do not collapse into
Source The origin context: connected account, imported archive, uploaded generic material, or derived internal source. One upload, one file, or one processing job.
Import session A bounded owner-visible intake event for one or more source items. The source itself.
Storage item The preserved original payload or file-level retained source. The semantic knowledge object created from it.
Processing job A resumable operation applied to source material, such as parse, map, enrich, summarize, or reprocess. Canonical truth or owner approval.

Source trust should affect automation aggressiveness. Strong structured sources can support more automatic baseline mapping. Weak generic, ambiguous, inferred, or unvalidated sources should produce lower-confidence mappings and more proposals. An owner-confirmed source may carry stronger trust, but it still does not silently become profile truth or public/shared material.

7.6 Connector Lifecycle

Each connected source should have an explicit lifecycle so Core can separate source behavior from individual item processing.

  1. Register: define source identity, account, access method, expected data types, privacy boundary, and owner consent.
  2. Initial backfill: import historical records in batches while preserving originals and writing canonical events.
  3. Incremental sync: track cursors, source item IDs, update timestamps, and processed ranges.
  4. Change detection: detect edits, replacements, moved items, and source-side metadata changes without losing the event-log record.
  5. Deletion/tombstoning: record deletions or unavailable source items as events instead of erasing Core's accumulated record.
  6. Duplicate prevention: use source IDs, content hashes, timestamps, and provenance to avoid creating duplicate truth.
  7. Review and reprocess: update source rules when real data shows better batching, extraction, or conflict handling is needed.

7.7 Processing Manifest Lifecycle

Processing manifests provide a lightweight starting queue before introducing a dedicated queue or workflow engine. They should live in Core system storage and move through explicit states.

  pending/
  processing/
  completed/
  failed/

original_ref: core://originals/audio/voice-note-001

This manifest model is not intended as a permanent high-volume queue. A real queue or workflow engine should be introduced only when processing volume, retries, concurrency, or observability require it.

7.8 Conflict and Source Precedence

When the same fact arrives from multiple sources, Core should not silently choose one without traceability. It should preserve each source input and create a current accepted interpretation with provenance.

fact: meeting_time
accepted_value: 2026-06-05T14:30:00+03:00
sources:
  - source: google_calendar
    value: 2026-06-05T14:30:00+03:00
    confidence: 0.98
  - source: email_invite
    value: 2026-06-05T14:00:00+03:00
    confidence: 0.82
resolution:

  selected_source: google_calendar
  resolved_by: enrichment_agent
  resolved_at: 2026-06-05T15:00:00+03:00

7.9 Adoption and Existing-System Transition

Core has two adoption tracks. Track 1 is a gradual start for a new owner who begins with an empty Core and grows it through controlled intake. Track 2 is a transition path for an owner who already has a custom personal information system, such as vaults, apps, histories, dashboards, agents, exports, source folders, or machine-local runtime state.

Track 2 should not import by folder size or convenience. It should begin with a system map by function, then assign core_disposition per area or item, then move through small owner-approved batches, then sequence functional areas by dependency, and only decommission old sources after parity and owner approval.

System map by function -> source-of-truth and disposition review -> bounded batch manifest -> peer technical screen -> checklist gate -> owner approval -> upload or hydrate -> retrieval/parity proof -> optional projection/index/export refresh -> decommission only after owner sign-off

Functional-area transition is a general PIOS pattern. Existing systems often contain owner-facing History, dashboard projections, communication threads, work objects, agent definitions, owner profile material, raw traces, data takeouts, saved-for-later sources, and code/repo folders. These should be mapped by their function in the owner's information system. Specific source paths, batch contents, and stage order remain implementation runbook details.

8. Event Model

Events are the canonical spine of PIOS. The complete append-oriented store of those events is the event log. An event describes something that happened, was captured, was inferred, was corrected, or was produced by the system.

Terminology rule: use event_log for canonical machine-readable event truth. Use History for the owner-facing readable journal or timeline surface derived from selected event-log records, structured updates, manifests, summaries, and retained artifacts.

The owner lives on one continuous timeline. Separation by device, app, AI service, source, runtime, or agent is a technical artifact, not how the owner experiences work and life. The event log preserves this full cross-source activity record with provenance; History is the readable owner-facing expression of that same underlying timeline.

8.1 Minimal Stable Envelope

The event schema should begin with a small stable envelope and allow metadata to evolve. The minimal required envelope should carry enough identity, version, time, actor, event type, idempotency, and logical reference information to interpret the event later. Source references, purpose, context, axes, and metadata enrich the event when known, but they should not block capture when unavailable.

The conceptual event envelope is storage-agnostic. It should use logical Core references such as core://events/<event_id> and core://originals/<source-object>. Physical keys, bucket names, S3 headers, filesystem paths, database row IDs, and cloud-provider write conditions belong to the storage adapter, writer manifest, or implementation-specific provenance, not to the conceptual minimum.

	  "occurred_time": "2026-06-05T14:30:00+03:00",

	  "primary_created_time": "2026-06-05T14:31:02+03:00",
	  "event_type": "email_received",
	  "actor": "connector:gmail",
	  "source": "gmail",
	  "event_ref": "core://events/01973f61-example-uuidv7",

In ordinary Core event records, the conceptual required fields are schema_version, event_id, idempotency_key, occurred_time, recorded_time, event_type, actor, and a logical event reference such as event_ref. source_original_time, primary_created_time, original_ref, purpose, context, axes, and metadata are required only when the event class or source contract needs them. Implementation-specific fields such as event_object_key, s3_write_condition, concrete bucket names, and filesystem paths document write realization and should live in writer metadata, manifests, adapter records, or implementation schemas rather than in the storage-agnostic conceptual minimum.

For example, the AWS-native event writer may materialize event_ref as a deterministic S3 object key such as events/year=2026/month=06/day=05/source=gmail/01973f61-example-uuidv7.json and enforce If-None-Match:* during the write. A local implementation may map the same logical event to a file path, SQLite row, or object-store key without changing the conceptual event meaning.

Use a time-sortable identifier such as UUIDv7 or ULID for event_id where practical. This keeps event IDs globally unique while preserving useful ordering behavior for logs, partitioning, and debugging.

8.2 Time Fields

Core must preserve the difference between when something originally happened in a source and when Core created the primary record. Source timestamps should not be overwritten with Core ingestion or promotion timestamps.

Field Meaning
occurred_time When the thing happened according to the source or best available interpretation.
source_original_time The original timestamp carried by the source item, such as an email sent time, calendar start time, message time, photo time, or external record time. This remains source provenance even after Core promotes the item into primary data.
recorded_time When Core recorded the event.
primary_created_time When Core created the primary Core record, knowledge projection, event record, or normalized object from the preserved source. For event records this may equal recorded_time; for later promoted knowledge objects it may be later than the original source capture.
primary_updated_time When Core last changed the primary record or projection without changing the original source item.
source_created_time When the source says the original item was created, if distinct from event time.
source_updated_time When the source says the item last changed.
processed_time When enrichment, extraction, or interpretation occurred. This normally belongs on processing events, enrichment manifests, derived projection records, or metadata about a processing step; it is not required on every ordinary event envelope.
idempotency_key Stable deduplication key derived from a source-native id when available; otherwise source, checksum, normalized source time, and a logical core:// original reference.
s3_write_condition AWS implementation metadata. Event-object writes should use deterministic object keys and If-None-Match:* semantics so retries cannot overwrite existing event objects. Non-S3 implementations should use an equivalent append/idempotency primitive.

In the AWS-native implementation, the primary event-write idempotency primitive is the deterministic S3 event object key with If-None-Match:*. DynamoDB may hold indexes, projections, workflow state, or secondary dedup records when useful, but it should not be required as the canonical event dedup mechanism.

Ordinary communication, raw captures, and low-level operational traces can still be preserved in the event log. They should not automatically become owner-facing completed-work History entries. 8.6 Correction and Reversion Events

Because the event log is append-oriented, rollback should be expressed as new events rather than destructive edits. If an ingestion, parser, enrichment run, owner correction, or agent action is later found to be wrong, Core should write a correction or reversion event that references the affected records and tells projections how to treat them.

{
  "schema_version": "event_envelope_v1",
  "event_id": "evt_01HX_STATE_REVERTED",
  "event_type": "state_reverted",
  "occurred_time": "2026-06-05T17:10:00+03:00",
  "recorded_time": "2026-06-05T17:11:03+03:00",
  "actor": "organizer_agent",
  "target_event_ids": ["evt_01HX_BAD_PARSE"],
  "target_object_ids": ["knowledge:summit_workshop"],
  "reason": "parser_bug",
  "reversion_scope": "derived_projection_only",
  "replacement_event_id": "evt_01HX_REPARSED",
  "projection_instructions": ["exclude_from_current_state", "rebuild_embeddings", "refresh_summary"]
}

Reversion events should not erase originals or hide the audit trail. They tell current-state views, indexes, summaries, retrieval systems, embeddings, and graph projections which records are superseded, invalidated, or excluded from active interpretation. The original evidence remains available for audit and repair.

9. Personal Glossary

The personal glossary provides the shared vocabulary that agents use to interpret the person's world consistently. It should remain simple at first: Markdown files, stable IDs, definitions, aliases, status, and relationships.

9.1 What Belongs in the Glossary

  • Terms unique to the person's life, work, family, companies, products, projects, and culture.
  • Nicknames, aliases, shorthand, and domain-specific meanings.
  • Roles, modes, relationship types, location types, privacy levels, source types, and event vocabulary.
  • Concepts that are broadly known but have a specific meaning in this person's world.

9.2 What Should Not Be Over-Captured

9.6 Visibility, Discovery, Access, and Rights

Sharing and public presence should not be governed by a single public/private flag. Cotton and Core should keep four outward-facing axes separate so agents can reason about exposure precisely.

The axes below are fine-grained outward representation controls. visibility describes who can see a specific representation, discovery describes whether it can be found, access_control describes the unlock mechanism, and rights describes allowed use. A coarse public access level must still map to a more precise visibility state such as public_unlisted or public_promoted.

Canonical private truth and outward-facing representation are not automatically the same object. A private profile, note, result, role page, or knowledge object may be copied, filtered, summarized, redacted, or transformed into a shared/public representation. 9.8 Ontological Connections

When concepts overlap across frameworks, Core should not force one false universal term. It should connect them:

Energy and resistance Helps match work to the right window. energy_low, energy_medium, energy_high, resistance_low, resistance_medium, resistance_high, thinking_mechanical, thinking_light, thinking_medium, thinking_deep

10.2 AI-Assisted Labels as Soft Semantics

AI-assisted labels are optional machine-applied cues that help filtering, retrieval, pattern detection, and proposal generation without becoming owner-confirmed truth. They are useful for imported material, chat records, notes, updates, events, and other objects where the system can infer helpful soft semantics but should not pollute the owner's confirmed taxonomy.

Rule Meaning
Assistance, not truth AI-assisted labels may support recall and analysis, but they do not carry the same authority as owner-created labels, glossary terms, roles, modes, or confirmed profile facts.
Toggleable by default Search, retrieval, and filtering surfaces should be able to include or exclude AI-assisted labels. The safe default is exclusion unless a surface intentionally opts in.
Controlled library Agents should apply only known AI-assisted labels from an owner-scoped library. If no suitable label exists, the agent proposes a new one instead of inventing ad hoc labels per object.
Archived, not erased Assignments should be archived rather than deleted so debugging, provenance, and past retrieval behavior remain explainable.
Promotion requires confirmation An AI-assisted label can become an owner-confirmed organizing label only through proposal, owner confirmation, and deduplication against existing labels.

10.3 Action Tags, Actions, and Skills

Action-oriented terminology should stay precise because intent labels, executable capabilities, and reusable playbooks behave differently.

Term Meaning Governance rule
Action Tag A lightweight intent label applied to an item, such as to read, follow up, draft, approval required, or archive. Guides handling and proposals; does not execute anything by itself.
Action An executable capability, tool call, workflow step, or system operation. Requires permissions, run logs, and appropriate owner visibility.
Skill A packaged playbook, method, or reusable capability definition behind one or more actions. Versioned and governed as system knowledge.

Action Tags are especially useful in capture and inbox flows because they let the owner communicate intent quickly without writing a full instruction. Automation may map an Action Tag to an Action only when a confirmed rule exists and any resulting autonomous execution is logged.

10.4 Context Tag Lifecycle

Roles and operating modes can attach to many object types as context tags. They help filter, prioritize, and retrieve information, but they do not grant or restrict access. Exposure remains governed by coarse access_level, relationship scope, visibility, discovery, access-control mechanism, rights, and sensitivity handling.

Removing a role or mode from an object should archive the context link rather than delete it. Reapplying the same role or mode creates a new active link while the old archived link remains as history. This preserves historical context, avoids silent rewriting, and lets History or audit views explain how an object's context changed over time.

10.5 Cross-Cutting Axes

Some classification concepts apply across domains, information areas, and object types. These should be modeled as cross-cutting axes rather than forced into life domains or folders.

As-needed attribute rule: metadata attributes should be used only when they add useful preservation, retrieval, interpretation, sharing-control, automation, or future reasoning value. Missing attributes do not mean false; they mean not relevant, not known, or not yet classified.

Axis schemas are intentionally extensible. Core is never locked into a final format: agents can later add new attributes to selected documents, events, profiles, Work Starters, or derived views when a new dimension becomes useful. New attributes may be backfilled or reprocessed across a source, project, time period, domain, or profile family without changing the canonical originals.

Axis Purpose Initial values or fields
Purpose / intent Explains why the item exists, why it was captured, or what outcome it supports. intent, desired_outcome, decision_supported, owner_relevance
Origin class Separates human-authored information from observed, imported, inferred, or generated information. authored, observed, imported, inferred, generated
Sensitivity Defines handling care inside Core before access or sharing decisions are made. normal, sensitive, guarded
Time relevance Describes whether the item is immediately actionable, currently active, reference material, or archival. immediate, active, reference, archival
Technical form Describes the material form agents must process, render, preserve, or transform. text, image, audio, video, structured_data, binary, code
Communication Captures communication-specific meaning without turning communication into a life domain. channel, direction, thread, participants, message_role
Location / movement Captures place, route, travel, and movement meaning across domains. place, route, trip, remote, physical, virtual
Preferences and habits Separates stable preferences, observed habits, routines, and constraints from one-off events. preference, habit, routine, constraint

Access level, relationship scope, exposure axes, and sensitivity remain separate. Sensitivity says how carefully Core should handle an item; access level says the coarse sharing tier; relationship scope says what kind of circle relationship is involved; visibility, discovery, access-control mechanism, and rights describe a specific outward representation.

10.6 Sensitivity and Guarded Handling

Sensitivity controls handling behavior inside Core. It does not create a separate information silo and it does not replace access control. A sensitive or guarded item can remain connected to events, sources, people, projects, and knowledge objects while still limiting projection, summarization, sharing, and action.

Sensitivity Meaning Agent behavior
normal Ordinary personal or operational information with no special handling risk beyond normal privacy and access rules. Agents may classify, summarize, index, retrieve, and project normally according to access level and owner preferences.
sensitive Private, personal, relationship-relevant, commercially relevant, identity-related, health-related, financial, legal, or operationally meaningful information where unnecessary exposure could cause discomfort, confusion, or harm. Agents may process inside Core, but should summarize carefully, avoid unnecessary exposure, preserve provenance, and prefer owner-private projections unless sharing is clearly intended.
guarded High-risk sensitive information or action context where misuse, leakage, hallucinated action, prompt injection, publication, or external exposure could cause material harm. Agents must not expose, publish, share, execute, or summarize into Circle, dashboards, exports, public surfaces, or owner-facing digests without explicit policy allowance, redaction, or owner approval.

Mark an item or action context as guarded when it contains or depends on secrets, credentials, tokens, private keys, environment files, personal identifiers, private owner-only context, sensitive health/finance/legal material, unpublished or publication-risk material, destructive or costly commands, external-facing decisions, untrusted inbound content, tainted attachments or URLs, or any action that requires explicit owner approval.

Guarded handling should apply proportional friction: minimize projections, redact where possible, keep raw details out of routine summaries, limit agent access by role and purpose, use stronger provenance, and route destructive, costly, publishing-related, externally visible, or otherwise high-impact actions through an explicit approval path. A notification, message, or ordinary update is not permission.

Downgrading guarded material to sensitive or normal should require explicit review or a clear policy rule. Agents should not downgrade guarded material only because it appears frequently or because an owner-authored or externally controlled message requests it.

10.7 Standards Mapping

Cotton should use established standards where they help interoperability, discovery, export, or agent understanding. Standards should be used as mappings, not as the owner of the model.

  • schema.org: useful for common public concepts such as Person, Organization, Event, Place, CreativeWork, Product, Action-like activity records, and other broadly understood entities.
  • Calendar and task formats: useful for calendar events, scheduling, invitations, reminders, and task exchange.
  • Contact and identity formats: useful for people, organizations, roles, addresses, communication handles, and relationship records.
  • Domain standards: useful when a vertical has strong existing vocabulary, such as health, finance, legal, academic references, travel, or product data.

When a standard conflicts with the personal meaning of a term, Cotton keeps the personal meaning in the glossary and stores the standard mapping as a secondary interoperability field.

10.8 Taxonomy Governance

Taxonomy values should be versioned and reviewable. Agents may propose new values, but canonical additions should include a reason, examples, related terms, deprecated alternatives, and expected usage rules. This prevents one-off tags from becoming accidental standards.

11. Life Domain Framework

The life domain framework is an initial LLM-familiar vocabulary for broad classification and retrieval. It is not a rigid ontology, not a fixed folder hierarchy, and not a limit on what Core can store. It gives agents common human language for first-pass organization while the glossary and context model handle overlaps, cultural nuance, and personal meaning.

Design rule: use common life-domain terms where they help, but do not force every item into exactly one domain. Information may belong to multiple domains or be better retrieved through roles, modes, people, projects, locations, sources, or events.

Mining finding: a heavily work-oriented corpus can make a work subject look like a missing life domain. Agents must distinguish domain_as_subject from domain_as_lived_data. A strategy document about personal data ownership belongs to Career/Work or My System; the owner's own identity facts belong to the Identity life domain.

11.3 Cultural and Linguistic Flexibility

Cultural concepts should be captured in the personal glossary with their native meaning, language, aliases, and approximate mappings. For example, the Finnish concept of sisu can exist as its own glossary term and connect to Personal Growth, Identity, resilience, perseverance, or other adjacent concepts without being flattened into one English equivalent.

11.4 Domain Evidence and Coverage

Domain lists should not be calibrated from one corpus alone. A work-chat archive may strongly evidence Career/Work, My System, Knowledge and Work Outputs, execution flow, and professional relationships, while barely representing Health, Family, Recreation, Possessions, or lived personal Finance. This is a coverage gap, not proof that those domains are less important.

The validation target is not a complete personal archive. The target is whether Cotton can organize plausible information from all major life areas without inventing new top-level concepts. Agents may use available owner sources, public schemas, public example records, synthetic data, fake profiles, and mock exports for this purpose.

12. Personal Context Dimensions

Context describes the personal situation around an event or knowledge item. The four core dimensions are roles, modes, location, and with whom.

Roles

Which capacity of the person was active?

Examples: founder, speaker, advisor, mentor, parent, Acme CXO, Summit workshop speaker.

Operating Mode

How was the person operating?

Examples: focused, reflective, strategic, exploratory, in delivery, traveling, learning, recovering.

Location

Where did this happen?

Physical, virtual, contextual, or inferred: home office, Metro Innovation Center, video call, walking outdoors, in transit.

With Whom

Who else was involved?

People, organizations, groups, audiences, participants, represented parties, mentioned people, affected people.

12.1 Role Type and Role Instance

Roles need at least two levels:

Level Example Use
Role type speaker, founder, advisor Broad retrieval and general reasoning.
Role instance acme_cxo, summit_workshop_speaker Precise retrieval tied to organization, project, time, or responsibility. 
role_instance:
  id: summit_workshop_speaker
  title: Speaker
  role_types:
    - speaker
    - workshop_facilitator
  related_project: summit_workshop
  related_organization: summit
  status: active

Roles may be expressed at several levels of precision. The system should allow retrieval at any useful level without forcing every event to carry all levels.

role_type: speaker
role_subtype: workshop_speaker
role_instance: summit_workshop_speaker
organization: summit

Multiple roles may be active in the same event. For example, a person may act as both founder and speaker during one workshop or as both advisor and product architect during one client meeting.

roles:
  - role_type: founder
    role_instance: northwind_cofounder
  - role_type: speaker
    role_instance: summit_workshop_speaker

12.2 Context Sources

Source Meaning

Inherited Context is inherited from an active meeting, project, calendar block, session, trip, or parent event.
Inferred An agent infers context from evidence and records confidence and provenance.

13. Functional Flow and Dynamic Scheduling

Cotton also covers the functional flow of personal operation: how goals become plans, plans become projects, projects become tasks and actions, and completed work becomes resolved knowledge, updates, and owner-facing History.

13.1 Operational Flow

The operational layer should preserve a clear chain from direction to completion:

Direction | v Goals | v Targets / key results | v Projects, initiatives, and responsibilities | v Work Starters | v Tasks, next actions, routines, and delegated agent work | v Done, resolved, archived, reflected, or reopened | v Updates, dashboards, patterns, summaries, learning loops, and updated Cotton guidance

This flow is not only a task-management model. It is an information model. A goal, target, project, Work Starter, task, routine, update, dashboard, completion record, reflection, and resolved outcome should all be represented as information objects with provenance, context, status, ownership, and links to events.

13.2 Update-First Owner History

When meaningful work reaches done, Core should create a structured completed-work update before relying on raw traces to reconstruct what happened later. This is the update-first owner History pattern.

Meaningful work reaches done | v work_completed event | +-- owner-facing work.completed update card +-- browser-readable detail page +-- machine-readable History manifest +-- retained source summary or artifact | v daily History | v weekly / monthly / quarterly / yearly summaries

Doing means planning, building, fixing, reviewing, validating, publishing, completing an assessment, or producing another bounded work outcome. Ordinary communication, raw chat, CLI output, cron logs, and session traces can be preserved in the event log, but they are fallback or provenance material until they produce an owner-relevant update, explicit decision, or later summary.

Owner-facing History should be owner-communication-first. Conversations, instructions, decisions, and work sessions where the owner was present, addressed, or directly involved carry more narrative weight than silent background activity. Background jobs, cron traces, connector syncs, and automated outputs still belong in the event log, but they normally enter readable History through completed updates, surfaced decisions, failures/recoveries, or later summaries.

Artifact Purpose Storage area
work_completed event Canonical event-log record that a bounded piece of work reached done. Events / core-events
Update card Short owner-facing progress item for dashboards, notifications, or activity views. Derived or application surface
Detail page Browser-readable explanation, main artifact preview, provenance, and verification context. Derived / update detail bundle
History manifest Machine-readable bridge from completed update to daily owner-facing History. Derived or system manifests
Retained source Markdown/YAML summary, source artifact, body file, or other durable evidence kept for audit and repair. Knowledge, originals, or task-specific storage

Markdown with YAML front matter can remain the retained semantic source of truth. HTML is the preferred presentation format for owner-facing update details, previews, and History surfaces, so the owner can read the result directly without opening raw files.

Daily History should read structured update manifests and related detail artifacts first. When a closed day already has structured update manifests, those update records are the owner-visible contract for completed work. Raw chat, CLI, Circle, connector, cron, and session traces remain available for recovery and gap filling, but they should not compete as equal narrative inputs unless a gap is detected.

History day assignment should be explicit. Daily History groups updates by history_day, not merely by the wall-clock time an update was published. A deployment may define a local history_day_cutoff, such as early morning, so late-night work belongs to the owner's lived previous day. A history_day_override may be used when an update is intentionally assigned to a different day.

Summary level Preferred input Rule
Daily Structured update manifests, selected event records, and fallback traces where needed. This is the normal level that reads source events and raw traces directly.
Weekly Closed daily summaries. Compose from the accepted daily record instead of reprocessing every raw source.
Monthly Closed daily summaries or closed weekly summaries, depending on the implementation. Stay bottom-up and preserve links back to the contributing days.
Quarterly Closed monthly summaries. Focus on direction, outcomes, patterns, unresolved themes, and changes in operating rhythm.
Yearly Closed quarterly summaries. Preserve the owner-facing yearly narrative without flattening the detailed lower-level evidence.

Daily History Template

Daily History is the most important and most concrete summary level. It should be treated as the normal owner-facing daily journal unit. Weekly, monthly, quarterly, and yearly summaries are derived upward from closed lower-level summaries and may mature over time, but they should not replace the daily structure as the operational anchor.

A daily History entry should carry a short summary field plus a structured rendered body. Current preferred sections include:

  • Summary: one short abstract of the day or History Brief.
  • Bottom line: concise owner-readable conclusion for the day.
  • Activity breakdown: topic or work distribution when enough structured updates exist.
  • Work by Topic: narrative grouping of the day's meaningful work.
  • Morning, Afternoon, Evening, Night: optional time-slot sections when the day is reconstructed from mixed source traces; night can cross the calendar boundary according to the deployment's day cutoff.
  • Decisions and directions: explicit owner decisions, policy changes, instructions, or direction changes.
  • Agent activity: background or system activity that matters but should not dominate the owner-facing narrative.
  • Open Threads / Next: unresolved items, carry-forward work, waiting decisions, or next actions.
  • Sources: provenance statement showing which updates, manifests, sessions, messages, or imported sources contributed.

Empty sections should be omitted rather than filled with placeholders. Older or imported daily records may use simpler sections such as Highlights or Notes, but new generated daily History should prefer the richer structure above when source signal supports it.

History Accordion Model

Rendered History should work as a progressive tree, not as a flat feed. Daily entries are always visible and are not drill-down accordions. Weekly, monthly, quarterly, and yearly cards show their own readable summary first; expansion reveals only the next child level below them.

Level Card behavior Expansion reveals
Yearly Readable yearly note and tags where useful. Quarters.
Quarterly Readable quarter note focused on direction, outcomes, patterns, and carry-through. Months.
Monthly Readable month note and optional topic tags. Weeks or daily fragments, depending on the implementation.
Weekly Readable week note and optional topic tags. Daily entries.
Daily Static header and always-visible daily body. Nothing hidden; the daily journal is already the leaf detail.

The UI should preserve this distinction in markup and behavior. Non-daily cards may use an interactive surface with aria-expanded, aria-controls, keyboard toggling, and a visible chevron. Daily cards should render as expanded static nodes. On touch devices, the accordion should avoid accidental expansion during scrolling by treating pointer movement beyond a small tolerance as scroll rather than tap.

Time-zoomable aggregated History is a first-class retrieval surface in PIOS. It lets agents and owner-facing interfaces navigate personal information by time, from broad periods down to specific days, updates, events, and originals.

This path supports context reconstruction and evolution tracking. 13.4 Dynamic Scheduling Model

The Dynamic Scheduling Model is the time-allocation and execution framework operating inside Cotton. It recognizes that real life rarely follows a fixed calendar plan. Agents should help match work to available time, decide what deserves attention in the current situation, and reorganize days or weeks without losing the underlying intent.

The model should be validated against existing planning concepts such as flexible time blocking, GTD-style next-action handling, calendar optimization, adaptive scheduling, and personal productivity workflows. PIOS should keep its own Dynamic Scheduling Model where those existing concepts do not cover agent-assisted context, the personal Core record, and live replanning across goals, projects, tasks, and real-world interruptions.

Mode distinction: operating mode describes how the person is operating in the moment; working mode describes the kind of effort a work item needs. Agents match a work item's working_mode to a time window where the person's operating mode, energy, location, and tools fit.

Attention rule: avoid static priority labels as the main planning mechanism. Use pinned when something should stay visibly at the top by owner choice. Otherwise, let Dynamic Scheduling order Work Starters and active work by purpose, commitments, timing, available window, energy, location, tools, dependencies, risk, and current context.

Concept Meaning Agent behavior
Available slot A real or predicted time window that can be used. Match work by duration, energy, location, context, urgency, and dependencies.
Pinned item An owner or agent-selected item that should remain visible at the top of a list without pretending it has universal priority. Keep visible until unpinned, completed, deferred, or superseded; still use scheduling context to decide when it is actually actionable.
Ordered Work Starter list A context-sensitive sequence of ready-to-start work for the current day, window, purpose, or review horizon. Reorder as context changes instead of relying on fixed high/medium/low priority labels.
Working mode The kind of effort a task needs. Separate deep work, admin, communication, recovery, errands, meetings, creative work, and agent-delegable work.
Dynamic replanning Calendar and task flow changes during the day or week. Move, split, defer, escalate, or delegate work while preserving the reason for the change.
Resolution The point where a task, project, or decision no longer needs active handling. Create completion events, update project state, write summaries, and feed learning back into patterns.

13.5 Base Week and Window-Based Planning

The Base Week is a realistic default rhythm, not a rigid calendar. It describes how a person's week usually flows, which working modes fit which windows, and how work should reflow when plans change.

Dynamic Scheduling should not only ask what is important. It should ask what important work fits the current window, energy, tools, location, commitments, and context.

  • Small pockets: useful for quick replies, approvals, simple admin, small follow-ups, and low-thinking responsibility work.
  • Deep windows: useful for writing, architecture, strategy, product thinking, workshop design, and complex decision work.
  • Voice or walking windows: useful for planning, rough drafting, reflection, idea development, and unresolved questions.
  • Travel windows: useful for reading, review, voice planning, light drafting, preparation, and reprioritization.
  • Review windows: useful for daily orientation, weekly planning, monthly direction adjustment, and yearly direction management.

Work Starter Ordering Heuristic

The ordered Work Starter list should be produced by a transparent fit heuristic, not by a permanent priority category. The Time Planner Agent can score candidate work against the current window and explain the ordering.

ordering_score =
  context_fit
  + commitment_or_deadline_pressure
  + owner_relevance
  + readiness
  + window_duration_fit
  + energy_fit
  + tool_and_location_fit
  + unblock_value
  + attention_signal_strength
  - switching_cost
  - resistance
  - risk_or_guarded_friction

Pinned items should remain visible, but they should not automatically outrank context-fit work when they cannot be acted on in the current window. The agent should preserve the ordering reason so a later review can explain why an item was surfaced, deferred, split, delegated, or skipped.

13.6 Today View

The daily entry point should be more than a task list. A PIOS-compatible daily view should combine orientation, pinned items, commitments, available windows, ordered Work Starters, pocket work, open loops, decisions, AI-preparable work, and carry-forward items.

Today with Agent - Orientation: what changed and what matters today? - Pinned: what should stay visible until addressed? - Commitments: what is fixed, promised, or deadline-sensitive? - Available windows: what time, energy, tools, and locations are realistic? - Ordered Work Starters: what is ready, logical, and context-fit now? - Pocket work: what small responsibility work fits short windows? - Decisions needed: what requires human judgment? - AI can handle: what can be drafted, organized, researched, or prepared? - Open loops: what is waiting, unresolved, blocked, or at risk? - Carry forward: what should be ready for tomorrow?

13.7 Continuous Adjustment Loop

The goal is to reduce heavy catch-up reviews by keeping goals, tasks, projects, Work Starters, and current ordering updated as part of normal operation.

Capture | v Clarify | v Classify | v Prepare | v Start | v Execute | v Reflect | v Adjust

Agents should use this loop to summarize new inputs, extract tasks from voice, detect open questions, suggest Work Starters, prepare drafts, break admin into small steps, requeue skipped work, surface decisions, and keep goals connected to daily work.

13.8 Hybrid Work and Attention Layers

Asynchronous and agent-driven work should use the same work model as human work. There should not be a separate AI-only work system just because an agent initiated or continued the work.

Layer Role Rule
Canonical work object The real state of a goal, project, Work Starter, task, routine, import session, processing job, or result. State changes here first.
Chat, Circle, or thread wrapper The communication surface where the work was requested, discussed, clarified, or followed. May summarize or route progress, but is not the canonical state store.
Update card The durable attention surface for completion, approval, failure, partial completion, or notable change. Created when the owner should notice or decide something.
History entry The owner-facing retained narrative after meaningful work reaches done or becomes historically relevant. Derived from events, update manifests, detail pages, and summaries.

Background states such as queued, running, waiting for input, blocked, retrying, partially completed, completed, failed, cancelled, and archived belong to the canonical work object or processing manifest. Chat can show compact progress. Updates carry durable attention. History preserves the completed or meaningful record.

13.9 Execution Anti-Drift Rules

Execution concepts should stay tied to mental modes rather than implementation modules. Do not introduce a new top-level execution concept unless it represents a genuinely distinct owner mode that cannot be expressed through direction, planning, projects, routines, tasks, Work Starters, or results.

Concept Anti-drift rule

Project An active one-off execution container, not a generic folder for everything related to a topic.
Routine A reusable execution structure. A concrete use of the routine is a run or execution instance.
Task / To-do Task is the general actionable unit. To-do is a standalone task context, not every task in the system.

13.10 Cotton Requirements for Flow Objects

Goals, targets, projects, Work Starters, tasks, routines, updates, dashboards, reviews, and completed outcomes should use shared metadata where possible:

Goal titles should be outcome-oriented. A useful convention is Verb + Outcome + Context. Avoid vague labels such as Goal: Storage; prefer titles that say what should become true and where, such as Establish Core source import lifecycle.

  • Status: proposed, active, waiting, blocked, completed, resolved, archived, reopened.
  • Classification: information area, information type family, stability layer, life domain, source type, and origin class.
  • Purpose and intent: intent, desired outcome, owner relevance, decision supported, and reason for the object to exist.
  • Context: role, mode, location, with_whom, project, and related event or source.
  • Time: created, due, scheduled, started, completed, reviewed, and relevant event links.
  • Responsibility: owner, human participant, agent participant, delegated agent, reviewer, and affected parties.
  • Handling axes: sensitivity, access level, relationship scope, time relevance, technical form, and communication or movement axis where relevant.
  • Operating fit: working_mode, time window, energy requirement, thinking depth, resistance, AI role, human decision needed, pinned state, and current ordering reason.
  • Outcome: done record, unresolved residue, next action, reflection, review update, or update to a persistent knowledge object.

14. Knowledge Layer

The knowledge layer contains living views of the person's world. These views are useful, editable, and readable, but they are not the only truth. They remain traceable to originals and events.

14.1 Knowledge Object Types

Knowledge object types are finer-grained living projections within the broader information-type families. For example, the people_relationships family can produce people, organizations, roles, and relationship records, while knowledge_work_outputs can produce projects, frameworks, topics, documents, decisions, and other reusable knowledge objects.

Use both axes when useful: information_type_family gives the broad Cotton-level meaning, while a knowledge object type gives the concrete living projection. Object types are instances or projections within type families, not replacements for the families.

Profile objects should be separated by subject and authority. The owner profile describes the human owner and belongs to Core Self/Myself. Contact profiles describe other humans and belong to My World with Circle relationship scope. Agent profiles describe delegated system actors and belong to My System with permissions, memory scope, source access, and governance.

14.2 Profile Families

Profiles are living knowledge projections, not raw originals. They summarize stable and slowly changing facts about an entity while linking back to source evidence and events. The profile subject determines the authority model.

Profile family Belongs to Authority and handling
Owner profile Core Self / Myself Describes the human owner: identity, preferences, values, roles, health, skills, operating context, goals, biography, and public/private self-representations. Agents may maintain it from evidence, but owner-authored facts and owner review override inferred updates.
Human/contact profile My World / Circle Describes another person or relationship: identity, aliases, contact methods, relationship scope, trust level, consent, roles, shared history, and interaction context. Inner Circle, Extended Circle, and Outer Circle scope determine how much context may be used or exposed.
Agent profile My System Describes a delegated system actor: purpose, role, skills, permissions, memory scope, source access, operating instructions, runtime targets, current state, and evolution history. Agent profiles are governed system information, not possessions.
Asset or possession profile My World / Possessions Describes owned or controlled things: devices, domains, accounts, licenses, warranties, property, and maintenance records. These may connect to agents or tools, but they are not actor profiles.

Stable profile facts should change only through correction or versioned update. Slowly changing attributes should keep source references, confidence, and last-reviewed dates. Dynamic activity should remain linked as events, updates, messages, tasks, or relationship records rather than overwriting the profile itself.

Profile truth is earned, not assumed. Imported records, chat statements, calendar facts, observed behavior, and agent interpretations may become evidence, but they should not directly become owner-profile assertions. The normal path is evidence to observed pattern or proposal, then owner confirmation or an established rule, then a versioned profile update.

Contradicted or stale profile facts should enter an explicit disputed or review-needed state instead of being silently overwritten. Resolution should preserve the earlier statement, new evidence, owner decision, changed fields, and validity window. This is especially important for preferences, roles, health, relationships, public biography, and any fact that may be true during one period but false later.

The owner profile should be split into four subtypes with different authority and exposure rules:

Owner-profile subtype Authority Use
Manual owner profile Owner-authored or owner-confirmed. Identity, biography, preferences, roles, values, boundaries, and stable self-description.
Public profile Owner-approved outward representation. Public biography, professional or public-facing knowledge bundles, website/RAG/twin source material, and shareable summaries.
Agent-context profile Delegated instructions and context that tell agents how to work with the owner. Operating preferences, communication style, standing constraints, working modes, escalation rules, and current context packets.
Observed profile Evidence-backed and usually proposed before confirmation. Inferred patterns, habits, preferences, role changes, recurring constraints, and behavioral observations with provenance and confidence.

These subtypes may share facts, but they should not silently overwrite each other. A public profile may be a curated projection of the manual profile; an agent-context profile may include temporary operating guidance; an observed profile should move through proposal, owner confirmation, or expiry before becoming durable owner truth.

14.3 Living Views Are Projections

A knowledge note may change as new events arrive. This is expected. The note should not pretend to be raw evidence. It should expose source references, confidence where relevant, and last evaluation time.

An artifact can be owner-facing and authored while still being a projection. Update detail pages, History manifests, dashboard pages, and generated indexes may be important owner surfaces, but their source-of-truth status should be explicit. If the retained Markdown/body file plus canonical event is the truth, the rendered HTML page or manifest is a projection. If the owner intentionally authors a document as the retained source, then that document can be canonical while later HTML, index, and summary versions remain derived.

14.4 Knowledge Compilation Layer

The Knowledge Compilation Layer is the process and structure by which Core turns preserved source material into maintainable, navigable knowledge. It is not the same as ingestion and not the same as retrieval. It is the ongoing agent-assisted work of creating and maintaining source notes, concept notes, entity notes, summaries, backlinks, indexes, contradictions, synthesis notes, and maintenance flags.

Originals preserved raw source material Events canonical records that something was captured, processed, updated, corrected, queried, or completed Knowledge Compilation Layer source notes, concept notes, entity notes, summaries, indexes, links, contradictions, synthesis, and maintenance flags Knowledge living Markdown/YAML projections governed by Cotton Derived rebuildable indexes, embeddings, graph exports, summaries, search projections, and analytical views System agent instructions, processing manifests, source rules, lint rules, retrieval policies, schemas, and maintenance routines

A wiki-style LLM-maintained knowledge pattern is useful inside PIOS because it shows how agents can maintain structured Markdown knowledge over time. In PIOS terms, raw material maps to Originals, source pages map to Knowledge source projections, concept pages map to Knowledge concept/topic/entity projections, indexes map to generated retrieval surfaces, backlinks map to Cotton relationships and graph projections, lint passes map to knowledge maintenance, and agent instructions map to System definitions and processing rules.

PIOS should not be reduced to an LLM-maintained wiki. PIOS also includes the canonical event log, source lifecycle, source trust, owner authority, privacy boundaries, access circles, dynamic scheduling, work execution, system governance, portability, and long-term personal context. Agents may maintain parts of the Knowledge Compilation Layer, but they do not own Core. The owner owns Core; agents operate under delegated rules, provenance requirements, permissions, and correction mechanisms.

14.5 Summary-First Knowledge Navigation

Each meaningful knowledge object, source projection, History summary, project page, person page, topic page, and major system object should include a concise agent-facing summary. The summary helps future agents decide whether the object is relevant before opening the full body.

The purpose is not only human readability. Summary-first navigation protects context windows, improves retrieval efficiency, and makes it easier to trace why an object was opened.

summary: >
  Defines Cotton as the context blueprint for PIOS, including how personal
  information is named, grouped, interpreted, related, and made useful to agents.
summary_updated_time: 2026-06-20T12:00:00+03:00
summary_confidence: medium

For longer notes, the YAML summary may be paired with a visible Markdown Summary section near the top of the object. Agents should normally read the index or summary first, open the full object only when the summary indicates likely relevance, and open raw originals only when provenance, verification, or deeper evidence is required.

14.6 Source Links, Backlinks, and Generated Indexes

Bidirectional links are retrieval edges for agents, not only human navigation. When a source projection contributes to a concept, both sides should preserve the relationship.

# Source note
concepts:
  - cotton
  - knowledge_compilation_layer
  - summary_first_navigation

# Concept note
sources:
  - source_2026_06_20_llm_wiki_assessment
  - source_2026_06_19_pios_master_doc

When links are uncertain, they should be marked as inferred or proposed rather than silently promoted to confirmed truth.

related_sources:
  confirmed:
    - source_123
  inferred:
    - source_456
  proposed:
    - source_789

Generated indexes are rebuildable retrieval surfaces. They summarize and point into canonical or primary records; they are not canonical truth. Useful indexes include knowledge/index.md, topic indexes, people indexes, project indexes, time-period History indexes, and system retrieval indexes. Each index should include object link, object type, status, short summary, main topics or tags, last updated time, and review state where relevant. Indexes should be regenerated when underlying knowledge objects change.

14.7 Knowledge Maintenance and Linting

Core should include a knowledge maintenance process. Its role is to keep the knowledge layer navigable, source-linked, and safe over time. Maintenance should normally produce flags, proposals, and review tasks rather than silently rewriting important knowledge.

14.8 Concept Promotion Rules

Agents should not promote every term, source mention, or claim into a durable concept note on first sight. A candidate concept may begin as an extracted term, tag, source-local claim, or proposed glossary item. It should become a durable knowledge object only when it proves useful enough to maintain.

  • It appears across multiple sources.
  • The owner explicitly names or uses it as important.
  • It is needed for retrieval, project continuity, or decision tracking.
  • It connects multiple people, projects, decisions, events, or sources.
  • It becomes part of the owner's operating context, glossary, work, or long-term understanding.
concept_status: candidate | stub | active | mature | deprecated
promotion_reason:
  - repeated_across_sources
  - owner_confirmed
  - needed_for_retrieval
source_count: 3
confidence: medium
needs_owner_review: false

A single-source idea can still be preserved, but it should usually remain attached to the source until it proves durable. Promotion is a maintenance decision, not a reflex.

16. Retrieval Model

Retrieval should depend on the question. No single method should be forced as the default for every query.

Retrieval Type Best For Initial Implementation
Keyword search Known words, titles, filenames, simple lookup. File/object metadata, extracted text, simple index.
Structured query Events by time, source, context, role, mode, project, participant. DuckDB over JSON/NDJSON/Parquet or Athena in AWS-native deployments.
Semantic retrieval Meaning-based discovery, fuzzy recall, natural-language questions. Embeddings and vector index/projection when justified.
Graph retrieval Multi-hop relationships, ontology reasoning, validation, relationship-heavy questions. Graph exports first; graph database only when operationally needed.
Hybrid retrieval High-value answers where completeness or verification matters. Run multiple retrieval paths in sequence or parallel.

16.1 Retrieval Routing

The Retrieval Agent decides which method to use. The default should not be always use vector search. The default should be to use the cheapest reliable route first, then expand only as needed.

  • "Show all documents created as Acme CXO in May" should use structured metadata first.
  • "What did I discuss with Jordan about the knowledge base?" should combine participant, topic, events, and semantic search.
  • "Find the original file from that June workshop" should use source/time/context lookup and original references.
  • "Validate how this project connects to my roles and organizations" may justify graph-style relationship traversal.
Route Best For Start With
Concept-first What do I know about this concept, topic, entity, or term? Concept/topic/entity index and summary pages.
Time-first Sequence, progress, recent activity, decisions, completion, before/after comparison, or how an idea changed. Year, quarter, month, week, or day History summaries.
Person-first Relationship history, discussions with a person, shared context, or communication patterns. Person profile, relationship notes, communication summaries, and linked History periods.
Project-first Project state, project history, decisions, open work, and completed outcomes. Project page, Work Starters, decisions, completed updates, and related History.
Graph-first How people, projects, concepts, decisions, sources, events, and organizations connect. Relationship edges, graph exports, and bounded entity neighborhoods.
Semantic/vector-first Exact passages, weakly structured material, fuzzy ideas, hard-to-name concepts, or large corpora. Scoped vector/search projection, then verify against source notes, events, or originals.
Hybrid Most high-value real questions. The cheapest high-signal route, then additional routes only as needed.

Time-first retrieval should be used when a question asks about sequence, change over time, progress, History, decisions, completion, context around a date or event, recent activity, idea development, before/after comparison, or weekly/monthly/quarterly/yearly review. The normal route is: identify likely time range, open the highest-level relevant History summary, narrow downward only as needed, follow links to daily entries, update cards, events, knowledge objects, people, projects, and originals, then answer with provenance and confidence.

16.2 Retrieval Result Requirements

Useful retrieval should return:

  • The answer or matched items.
  • Source references to originals, events, and knowledge notes.
  • Confidence where interpretation is involved.
  • The retrieval method used.
  • Any gaps or conflicts found.

Retrieval answers should also preserve the route and evidence chain where appropriate. The owner-facing answer does not always need to show the full trace, but the system should be able to keep it for debugging, trust, and future improvement.

retrieval_trace:
  route:
    - time_first
    - concept_followup
  opened:
    - history/2026/06/index.md
    - history/2026/06/week-25.md
    - knowledge/concepts/cotton.md
    - events/2026/06/20/event_123.json
  not_opened:
    - raw_transcript_originals
  reason_not_opened: "weekly and concept summaries were sufficient"
  confidence: medium
  gaps:
    - "No owner-confirmed decision record found for final naming date."

16.3 Context Packets and Retrieval Logs

Retrieval should be treated as deterministic context assembly before LLM generation. The Retrieval Agent produces a context packet, and the model uses that packet; the model should not be responsible for silently deciding which long-term memory to inspect.

Retrieval tier Purpose Default bias
Immediate continuity Keep the current interaction coherent. Recent turns and active view.
Direct context Use recent material from active roles, modes, channels, projects, or source filters. Current context before global search.
Consolidated summaries Avoid rereading raw history when daily, weekly, or topic summaries answer the question. Summaries before old raw traces.
Canonical knowledge Bring in stable notes, profile facts, glossary terms, project records, and decisions. Confirmed or owner-authored knowledge before inferred material.
Entity neighborhood Expand through related people, organizations, projects, places, topics, and terms. Bounded graph expansion.
Scoped vector search Handle paraphrase, fuzzy memory, and broad questions only after scope is constrained. Filter-first; no unscoped search across years by default.

Every meaningful context assembly should create a retrieval log or equivalent trace that records active filters, cue interpretation, policy version, selected item IDs, selected tiers, rationale, token budget, and whether semantic/vector search was used. Deep recall beyond the recent operating window should be triggered by explicit time intent, selected entities, selected context, or recent-retrieval failure, and the reason should be logged.

16.4 Vector Retrieval as a Projection

Vector retrieval should be added as a projection when structural navigation is no longer sufficient. It should not replace the knowledge structure, the event log, or preserved sources.

  • Add vector retrieval when the number of knowledge objects grows beyond what summary-first navigation can handle efficiently.
  • Add it when exact passages are frequently needed from long documents, transcripts, dense source material, or large corpora.
  • Add it when the owner asks for weakly named or semantically fuzzy information that concept and History indexes do not narrow reliably.
  • Keep embeddings, vector stores, and search indexes rebuildable from retained sources and knowledge objects.

Recommended rule: build the wiki-like knowledge structure first, then add vector retrieval when the structure no longer answers enough queries efficiently. When chunking is needed, prefer structure-aware chunks based on Markdown headings, sections, object boundaries, source notes, History items, and event-linked units rather than arbitrary token-count slices.

16.5 Storage Is Not Memory

A folder of files is not memory. A vault becomes useful memory only when Core can preserve source material, maintain structured knowledge, navigate summaries, trace claims to evidence, update indexes, detect inconsistencies, and retrieve context across time.

Memory-like behavior emerges from durable preservation, event logging, structured knowledge projections, summary-first navigation, time-zoomable History, bidirectional relationships, retrieval routing, maintenance and linting, provenance, and correction. PIOS should avoid describing static storage alone as memory.

17. Agent Architecture

Agents are not external tools bolted onto the data store. In the Core Self model, trusted AI agents are part of the person's inner circle and operate from Core as their source of context and truth. Cotton adds an important rule: agents themselves are information and must be documented, governed, versioned, and retrievable like other important system entities.

17.1 Agents as First-Class Information

Each persistent agent should have an explicit profile that is human-visible, agent-readable, and operationally connected to the files and runtime settings that define the agent. The profile should include purpose, role, responsibilities, skills, permissions, memory scope, source access, operating instructions, current state, and evolution history.

Agent profiles may be represented through standard Markdown-style files such as AGENTS.md, agent.md, memory.md, skills.md, and implementation-specific runtime documents. Core should preserve the canonical profile and let the Coordinator Agent synchronize deployment-specific copies.

Runtime instruction files are governed system artifacts, not hidden local preferences. Files such as AGENTS.md, tool-specific instruction files, skills, memory files, MCP configuration, bridge configuration, and agent runtime policy should have canonical source copies, deployment targets, sync status, owner-readable explanation, version history, and rollback path.

17.2 Workflow Packages and Runs

Reusable agent and automation work should be packageable without turning the package into personal truth. A workflow package defines what a repeatable workflow does, which inputs it needs, which outputs it may create, which triggers it supports, which permissions it requires, and which safety constraints apply. Published workflow packages should be versioned; updates create new versions rather than silently changing prior runs.

Concept Meaning Governance rule
Workflow package Reusable, versioned definition of a workflow, skill bundle, or result-producing process. Declare inputs, outputs, permissions, triggers, parameters, and safety constraints.
Installed workflow The owner's configured instance of a package. Owner may enable, pause, configure, duplicate, update, or remove it.
Workflow run One execution instance with input snapshot, steps, outputs, status, errors, and logs. Every durable output preserves provenance to the package, installed workflow, run, inputs, agent, and time.
Truth boundary Workflows can produce drafts, artifacts, updates, proposals, AI-assisted labels, and run logs. They must not silently mutate owner-profile truth, rules, glossary naming, sharing state, or other governed canonical facts.

17.3 Owner-Facing Agent Profile Surface

Agent profiles should have an owner-facing HTML/UI representation, not only hidden runtime files. The owner should be able to inspect and manually edit profile fields while Core preserves the underlying canonical Markdown or structured source.

Surface element Requirement
Profile view Show purpose, role, responsibilities, skills, permissions, memory scope, source access, current state, and active runtime targets.
Editable fields Allow owner-approved edits to profile text, role descriptions, permissions, skills, and operating instructions.
Underlying files Show which canonical files and runtime files are affected, such as AGENTS.md, agent.md, memory files, and skill definitions.
Sync state Show whether runtime copies are current, pending sync, failed, or intentionally different from the canonical Core profile.
Change history Track created time, modified time, modified by, change reason, previous version, and affected runtime targets.

17.7 Agent Work Methodology

Agents should use a plan-first method for complex work. A lightweight fix does not need ceremony, but changes that touch shared systems, APIs, data integrity, security, multi-agent behavior, source ingestion, owner-facing surfaces, or multiple components should begin with an explicit implementation plan.

Plan element Requirement
Current state Describe what works, what is broken or missing, and what source evidence was read before changing the system.
Scenarios Enumerate normal flows, edge cases, failure modes, concurrent cases, security cases, and data-integrity cases where relevant.
Implementation requirements State what should change, what should not change, dependencies between changes, and the order of operations.
Validation criteria Define observable pass/fail evidence. "Seems to work" is not enough for shared systems.
Risk and rollback Identify blast radius, guarded actions, owner approvals needed, and a recovery path if the change fails.

Agent-created implementation plans are not automatically PIOS Plan objects. They are operational work artifacts. If a plan becomes owner-facing strategy or execution structure, it can be linked or promoted into the functional Goal/Target/Plan/Project model.

Technical Loop Prevention Protocol

Every agent-to-agent or agent-to-harness message that can trigger another agent should carry loop-control metadata. At minimum: conversation id, thread id, parent message id, origin actor, current hop count, maximum hops, collaboration grant id when relevant, turn budget, execution job id when relevant, and whether the message is allowed to trigger replies.

The harness should stop reply generation when a message arrives from a background agent-post path, when no owner-approved collaboration grant exists, when the hop or turn limit is reached, when the same actor repeats without new owner input, when a guarded action would be required, or when the accumulated execution/update budget is exhausted. This keeps agent communication useful without allowing API-token cascades or autonomous peer loops.

This pattern keeps the communication layer replaceable. A Circle app may be the collaboration interface in one deployment, while another deployment may use Slack, Matrix, email, Telegram, a native app, or a custom web dashboard. The Core API should accept structured communication envelopes from any harness, preserve raw source references where appropriate, and let Core organize the resulting information automatically for later recall.

Communication harnesses should also keep a security boundary between untrusted inbound content and the reasoning/action core. Raw messages, attachments, and external links should be authenticated, normalized, assigned trust metadata, and passed downstream as typed envelopes. Natural-language text from a sender should not be allowed to modify system policy, hidden instructions, tool permissions, or owner-only capabilities without explicit identity, policy, and approval checks.

19.2 Core Managed Owner Isolation

Core Managed can share AWS infrastructure, but it must not collapse different people's information into one undifferentiated SaaS data space. Each owner needs an explicit logical Core with clear boundaries around originals, events, knowledge, derived projections, and system definitions.

Per-owner data boundaries are explicit because ownership, export, deletion, and takeover matter. A person must be able to understand what is theirs, export it, delete it, migrate it, or take over operation of it without depending on hidden cross-owner coupling.

The managed service is fiduciary infrastructure for the owner. It may operate shared systems, but it should behave as a steward of owner-held information rather than as the conceptual owner of the data.

Preferred owner-isolated AWS layout: Core Managed should start from one physical S3 bucket per owner, with the five Core data zones represented as top-level prefixes. This keeps the owner boundary clear without creating unnecessary bucket sprawl for individual owners. Multi-bucket-per-owner layouts can remain available for exceptional sovereignty, regulatory, or operational cases.

This is a deliberate simplicity tradeoff for the owner-isolated pilot, not a claim that S3 has native per-prefix defaults. Separate buckets per data zone would make bucket-level default KMS keys and bucket-level default Object Lock retention simpler to configure, but they would also create more physical resources per individual owner. The single-bucket profile is acceptable only when write paths, tests, and bucket policies enforce the required prefix behavior. A high-assurance deployment may choose separate buckets per zone when native bucket-level controls matter more than physical-resource simplicity.

Shared physical S3 buckets with owner-scoped or service-tenant-scoped prefixes and strict IAM/KMS boundaries. Use one bucket per owner for the owner-isolated starting profile. Use shared service buckets only when the service-scale profile proves the need and can preserve owner boundaries, prefix isolation, key separation, lifecycle controls, export, deletion, and takeover semantics.

Conceptual invariant: the five Core data zones are per-Core and per-owner concepts. They are not AWS buckets. In the AWS-managed implementation, the preferred owner-isolated starting layout is one bucket per owner with data-zone prefixes; tables, indexes, KMS keys, retention rules, and namespaces can still vary as long as the owner's boundaries remain clear and exportable.

Single-bucket enforcement requirement: one owner bucket is simpler for individual owner boundaries, but per-zone KMS keys and selective Object Lock cannot be treated as bucket defaults. Writers must set the correct SSE-KMS key by prefix, writes into originals/ and events/ must set per-object Governance retention with a finite retain-until date and required checksum, and bucket policy must deny protected-prefix writes when the required KMS or Object Lock headers are missing or wrong. A short retention window is only near-term accident protection; long-term canonical-truth integrity needs an owner-ratified retention, backup, legal-hold, or tamper-evident manifest policy before bulk personal-data migration. Indefinite preservation should use legal hold, and s3:BypassGovernanceRetention should be limited to the emergency operator role.

Protected-prefix write enforcement must cover ordinary object creation and multipart object creation. In S3 bucket policies, multipart initiation is authorized through s3:PutObject semantics rather than a separate s3:CreateMultipartUpload action. Large originals such as drive exports, audio, photos, video, and data takeouts may use multipart uploads; writers must still supply the required SSE-KMS and Object Lock headers when starting the protected object write, and bucket policy must deny s3:PutObject when those headers are missing or wrong. Upload parts alone are not enough to establish the object's final encryption or retention posture.

Object Lock retention policy should be expressed as a floor, not an exact match. Bucket policy should deny protected writes whose remaining retention is below the owner-ratified minimum, while writers should set a small buffer above the minimum, such as retention days plus one day, to avoid clock-skew or rounding failures. Longer retention should remain allowed for critical originals unless a separate governance rule forbids it.

Per-zone KMS keys do not create access isolation by existing. Cross-zone isolation requires scoped key policies or scoped IAM grants that allow each role to encrypt, decrypt, and generate data keys only for the zones it is allowed to use. The default account-root key policy keeps keys manageable and permits IAM delegation, but by itself it provides administration and rotation separation, not the derived cannot decrypt originals boundary.

Lifecycle policies must be written with Object Lock behavior in mind. Lifecycle transitions may move locked versions to cheaper storage classes when allowed, but expiration cannot delete Object-Lock-protected versions until retention permits deletion. Do not depend on lifecycle expiration to erase locked originals or events during an active retention period.

19.4 Processing and Long-Running Work Rules

  • Lambda is a bounded worker: use Lambda for event-triggered processing, small extraction jobs, routing, manifest updates, and short agent tasks. AWS Lambda has a maximum invocation duration of 900 seconds, or 15 minutes, so jobs that may exceed that limit should be split into resumable steps or moved to a longer-running worker. Do not design Lambda as a persistent loop or always-running agent.
  • Orchestrate multi-step work: use Step Functions, SQS, EventBridge, or processing manifests when work needs retries, state transitions, fan-out, human approval, or resumability.
  • Move long-running execution out of Lambda: use Fargate, Batch, EC2, Glue, or a managed agent-worker service when jobs require long runtime, large memory, specialized dependencies, persistent sessions, or continuous polling.
  • Keep Bedrock optional: Bedrock or another LLM service can support extraction, reasoning, enrichment, summarization, and retrieval synthesis, but it is not the Core storage layer or the source of truth.
  • Separate control state from truth: DynamoDB, queues, workflow engines, and caches may track processing state, current projections, and operational indexes. They should remain rebuildable from canonical retained sources unless explicitly promoted through governance.

19.6 AWS Retrieval Rules

  • Keyword retrieval: start with object keys, object metadata, extracted text, or a generated index. S3 object listing is key/prefix based and should not be described as full content search.
  • SQL retrieval: use Athena over extracted event and metadata projections, preferably partitioned and converted to Parquet for recurring analytics.
  • Semantic retrieval: add S3 Vectors or another vector projection only when semantic search becomes frequent enough to justify it.
  • Graph retrieval: generate graph exports first; use Neptune only when multi-hop graph traversal becomes operationally important enough to justify its standing cost.
  • Hybrid retrieval: route based on the question, and return sources, method used, confidence, and unresolved gaps.

Semantic retrieval clarification: semantic retrieval is part of the PIOS retrieval model from the start, but a dedicated vector index is an implementation choice. Early drafts treated S3-native vector capability as effectively default; this master document separates the retrieval capability from the cost-bearing vector infrastructure so Core can enable it when real usage justifies it.

19.7 AWS Cost Caveats

AWS costs should be treated as illustrative until measured with real data. The cost model should include or explicitly exclude S3 storage and requests, Lambda requests and compute, Athena scanned data, Glue jobs, S3 Vectors storage and queries, Bedrock tokens, SQS/EventBridge/Step Functions, CloudTrail data events, KMS, monitoring, data transfer, and backup/replication.

Audit choices affect even an empty pilot. Multi-region CloudTrail, log-file validation, and S3 data events are deliberate guardrail choices, not free defaults. If a pilot has a monthly cost guardrail, document whether it uses multi-region plus data events from the start or stages them by gate.

Important wording: prefer "pay-per-use where possible" over "no fixed costs." Optional services can introduce baseline or idle costs. Versioning is not a backup strategy by itself.

19.8 AWS-First Roadmap

The AWS roadmap should follow explicit gates rather than treating foundation as one undifferentiated deployment. A practical sequence is: local validation, cost/readiness review, account preflight, CDK bootstrap, empty stack deploy, post-deploy validation, synthetic object dry run, synthetic upload test, and only then owner-data migration. Each gate should state what it authorizes and what it does not authorize.

The empty AWS deployment should also produce a reusable Core Template: versioned CDK source plus a synthesized CloudFormation template for the empty owner Core. The template must keep owner-specific values as parameters or manifest-bound context, including account, region, owner id, owner slug, bucket suffix, human trust principals, retention posture, and KMS posture. A later migration or takeover uses a Core Bundle for data hydration; deploying the template alone must not imply permission to upload synthetic or real owner data.

The synthetic protected-write gate should be explicit and repeatable: first generate a dry-run request plan, then obtain separate owner approval, then write only synthetic objects through the same protected path that real originals and events will use. After upload, validate with object metadata reads that the objects exist, use the intended zone KMS keys, carry Governance retention, preserve synthetic markers, and have the expected retain-until date. Replaying the same deterministic write should fail with an idempotency/precondition error such as PreconditionFailed from If-None-Match:*; that failure is a success signal because it proves retries do not overwrite or create duplicate canonical objects.

The first owner-data migration proof should be a reviewed slice, not a bulk import. A good first proof is one completed-work update set: one selected source artifact, one original, one canonical event, one processing manifest, pointer/index records where used, one owner-facing update detail artifact, and one History or retrieval proof that resolves back to the event and original. The slice should include sensitivity classification, source and Core timestamps, deterministic keys, idempotency preview, validation commands, and separate owner approval before upload.

Reference pilot evidence: the AWS pilot has been used to prove the implementation shape behind this specification: guarded upload tooling, S3 If-None-Match:* event/object idempotency, retrieval verification for narrow owner-data slices, Core Bundle export tooling, local bundle validation and restore proof, destination hydration into a separate proof Core, additive KMS Option B key-use grants, governed large-object multipart ingestion, manifest-driven reusable owner-template configuration, and AWS Backup S3 restore proof for a selected bundle. These are reference implementation learnings, not the canonical operational log for every Core deployment. Live pilot status belongs in the implementation repo README and current runbooks; this master document records the architecture pattern and the gates that a Core Managed deployment should satisfy before broader migration.

Governed shadow period: when a live flow is validated through write-through, the normal producer path must be wired into the shadow and evidence path rather than depend on a separate optional command. The canonical side publishes first; the destination remains a bounded shadow copy; each accepted write receives idempotency/replay and artifact verification; and reconciliation compares the shadow records with an independent producer ledger. Producer-publication time governs coverage reconciliation, while destination-acceptance time governs destination write limits. A clean bounded period informs a later owner cutover decision but never performs cutover by implication.

Owner-data migration should follow the adoption model in §7.9. Track 1 starts with controlled first intake for a new owner. Track 2 transitions an existing custom personal information system through functional system mapping, core_disposition, bounded reviewed batches, dependency-aware sequencing, and decommission-after-parity. 20. Core Self-Hosted Implementation

Core Self-Hosted implements the same PIOS/Core model on a portable VM or server stack. It exists so a complete Core can run outside the managed AWS service without changing the underlying information architecture.

In the self-hosted profile, the equivalent of a Core Template is a versioned, data-empty image-root or install composition: for example a VM image, install script, container composition, or configuration bundle that creates the same storage zones, services, backup rules, access controls, and processing structure before any owner data is hydrated. The image-root should be built from an allowlist, validated by a hygiene gate, and packaged only when it is provably free of owner data, deployment identifiers, and key material.

Current implementation stance: build AWS Core Managed first as the concrete reference implementation, but maintain a Core Self-Hosted implementation contract in parallel. The self-hosted contract should capture the portability requirements that AWS work must keep satisfying: five Core Data Zones, atomic no-overwrite event writes, retained originals, provider-neutral Core Bundles, local restore, History rendering, generated indexes, key custody, backup/restore, and owner takeover. Self-hosted validation should progress in stages: local bundle hydration, booted VM proof, protected-ingestion parity, hardened key provider, repeatability, and only then broader migrated owner data.

Transition patterns must stay profile-agnostic. AWS-specific controls such as S3 Object Lock, KMS crypto-shred, and a governed large-object multipart role should map to self-hosted equivalents rather than becoming PIOS requirements. A self-hosted Core needs its own documented large-file ingest path, immutability or append-only mechanism, key custody and erasure path, remediation/tombstone lifecycle, and restore proof before it handles broad migrated owner data.

20.1 Self-Hosted Template and First Boot

A self-hosted Core Template should initialize an empty Core from a self-hosted Provisioning Manifest. The manifest binds owner identity, host paths, key references, service ports, backup profile, optional bundle reference, and authorization gates. Hydration, connector start, scheduler start, and other operational actions should default to false and require explicit approval.

First boot should fail closed. It should refuse to initialize a non-empty Core root unless a governed recovery path is being run, create the five Core Data Zones before data hydration, record a health-check or initialization record in system/, and keep key material outside the Core root. Core may record key ids, fingerprints, and provider references, but not private key material or cloud secrets.

The key-provider abstraction should be explicit. A local development proof may use file-backed keys with strict filesystem permissions, while production self-hosted deployments should use a hardened provider such as an encrypted key store, hardware-backed keys, or a managed local secret service. The portability rule is the same as Core Managed: Core records custody metadata and fingerprints; the key provider holds the secret material.

20.2 Selected Initial Stack

20.3 Garage Notes

Garage is suitable for the self-hosted profile because it provides an S3-compatible API, is self-contained, has low resource requirements, and can later expand from a single server into a multi-node deployment. It also supports block-level deduplication and optional compression.

Single-node limitation: a single Garage node is still one machine. Garage's resilience and replication benefits become meaningful when data is distributed across multiple nodes or zones, so the single-VM setup must always have independent off-site backup.

21. Portability, Security, and Governance

21.1 Portability Requirement

A complete Core must be exportable, recoverable, and runnable outside the infrastructure provider currently hosting it. The export should not merely be a folder of owner files. It should be a reconstructable personal information core.

A Core export should include:

  • Originals
  • Canonical events
  • Wiki knowledge
  • Personal glossary
  • System definitions
  • Connector configuration, excluding secrets where necessary
  • Derived data where useful
  • Rebuild instructions
  • Deployment configuration

21.1.1 Core Template, Core Bundle, and Provisioning Manifest

Portability is clearer when Core transfer is split into three artifacts:

Artifact Role Rule
Core Template Parameterized infrastructure structure, such as CDK source and a synthesized CloudFormation template for an empty Core. The template carries the golden structure only: storage zones, keys, policies, roles, indexes, audit logging, and guardrails. Owner/account-specific values stay parameterized.
Full Core Export Bundle The owner's complete exportable data payload for transfer, rebuild, or takeover: originals, events, knowledge, selected derived data, system definitions, manifests, checksums, and rebuild instructions. The full bundle is reconstructable and key-agnostic. Destination deployments re-encrypt the imported data to destination keys instead of trying to move source keys. Scoped Core Bundles, such as project or person bundles, use the same pattern at smaller scope.
Provisioning Manifest The build spec that binds a template version to an owner, account, region, security posture, human trust principals, expected outputs, and optional bundle hydration. Every deployment-specific value belongs in the manifest or deploy parameters, not hidden in the template. The completed manifest should be retained in the Core system/ zone and in operator records.

A Provisioning Manifest should include, at minimum, the Core instance identity, AWS account and region, owner id and slug, bucket suffix or equivalent namespace, template version, retention policy, retention write buffer, Object Lock mode, KMS posture, protected prefixes, human principal references, expected stack outputs, data-bundle reference, and explicit authorization gates. Authorization fields should default to false: preflight, bootstrap, deploy, hydrate, and cross-account transfer must be separately approved. In Core Self-Hosted, the same manifest pattern should bind host paths, service ports, key-provider references, backup profile, optional bundle reference, and first-boot authorization gates.

The rebuild model has three common scenarios. A new empty Core deploys the Core Template with a completed manifest and stops before data hydration. A disaster-recovery rebuild recreates the empty structure, restores a full Core Export Bundle or backup, writes protected objects through the compliant write path, and rebuilds indexes and derived projections. A cross-account, cross-region, or takeover transfer exports a full Core Export Bundle from the source, deploys a new empty destination Core, hydrates the bundle with destination KMS keys, rebuilds projections, validates parity, and decommissions the source only after owner sign-off through the deletion and crypto-shred governance path in §21.6.

Parity validation should be explicit. At minimum, destination object counts and checksums should match the export manifest, protected objects should have the intended destination encryption and retention posture, provenance links should resolve from knowledge to events to originals, owner boundaries should show no cross-owner leakage, and selected retrieval paths should produce the expected evidence chain.

Tooling status boundary: before transfer or rebuild claims, an implementation should hold a current-state full Core Export Bundle proof covering export, checksums, restore, and selected retrieval paths. Rebuild and transfer should be treated as partially proven until destination hydration, retention posture, DR behavior, key custody, and parity validation have been tested for the intended implementation profile. Live pilot status belongs in the implementation repository README and current runbooks; this master document records the architecture pattern and the gates a Core deployment should satisfy.

Transfer caveats: cloud infrastructure is not one clonable disk image. KMS keys are account- and region-bound and cannot be exported, so transfer requires decrypting with source keys and re-encrypting with destination keys. S3 bucket names are globally unique, so names and suffixes remain deploy-time parameters. Object Lock retention is per object version; replication or hydration must preserve or reset retention intentionally. Account IDs, role ARNs, MFA flows, and external human principals must be revalidated at the destination. DynamoDB indexes, embeddings, graph exports, and other derived projections should be regenerated unless explicitly carried as selected derived data.

21.2 Backup and Recovery

Versioning is not backup. Every Core deployment must have independent recovery, restore testing, and exportability.

Deployment Minimum Protection Model
Core Managed AWS S3 versioning, AWS Backup or equivalent, cross-region replication/export where justified, protected credentials/keys, and periodic restore testing.
Core Self-Hosted Provider snapshots, encrypted off-site Restic backups, protected Garage/WireGuard/Restic credentials, and periodic restore testing.

AWS Backup Vault Lock is the backup-side WORM control that can complement S3 Object Lock on retained originals and events. It should be considered when backup retention needs tamper resistance rather than only near-term recovery protection. It is not implied by using AWS Backup; it requires a separate retention/compliance decision, cost review, and restore/deletion governance.

21.4 Privacy and Access

Access levels should align with the Core and Circle model:

Level Meaning
Private Core Self only: owner and trusted private agents.
Privately shared Inner Circle and selected Extended Circle members or agents.
Public Outer Circle or intentionally published information.

These access levels are coarse sharing tiers. They do not by themselves specify whether a public representation is unlisted or promoted, whether it is indexed, what unlock mechanism is required, or what reuse rights apply. Those fine-grained outward controls belong to the visibility, discovery, access-control, and rights axes in §9.6.

Sensitivity is a separate handling axis, not an access level. A guarded item may still be private inside Core Self, but it requires stricter review, stronger provenance, safer agent permissions, and more careful promotion before it appears in Circle, dashboards, exports, or owner-facing summaries.

Approval is separate from notification. If an item or action is guarded because it is sensitive, destructive, costly, publishing-related, externally visible, irreversible, prompt-injection-tainted, or otherwise high-risk, agents should use a request-scoped approval or verification path before acting or exposing it. A push notification, dashboard entry, Circle message, or ordinary History update is not sufficient permission.

21.5 Ingestion Gate Governance

Owner-data ingestion and existing-system transition should use a three-tier gate so technical review, checklist readiness, and owner authority do not collapse into one agent action.

Gate Meaning Cannot do
Peer technical screen An agent or operator verifies the candidate set, checksums, timestamps, sensitivity screening, planned keys, event mapping, and absence of excluded material. Does not authorize upload, retention posture, broad migration, DynamoDB projection writes, connector sync, or source decommissioning.
Checklist approval The batch satisfies the governing review checklist: scope is bounded, stop conditions are closed, retention/remediation posture is decided or explicitly blocked, and the planned proof is coherent. Does not replace owner approval for the exact manifest and object/event plan.
Owner approval The owner approves the exact manifest, retention posture, remediation posture, projection/export choices, and upload boundary. Does not authorize anything outside the approved batch or future broad migration unless explicitly stated.

This is a fail-closed gate: unresolved authority, sensitivity, retention, remediation, or source-of-truth questions keep the upload blocked by default. A useful review result may be technical_screen_status: pass while checklist_review_status: fail_pending_owner_decision and upload_authorized: false. That is not a failure of the review; it is the correct expression of owner authority and fail-closed ingestion governance.

Golden templates, VM images, install payloads, and reusable deployment packages are also governed artifacts. Before distribution or reuse, they should pass a hygiene check proving they contain no owner data, deployment identifiers, private key material, cloud credentials, or environment-specific secrets.

21.6 Fiduciary Service Governance

Core Managed should be governed as fiduciary infrastructure for owner-held information. The service may manage infrastructure, processing, backups, connectors, indexes, and applications, but the owner remains the primary authority and conceptual owner of the information.

Obligation Requirement
Owner authority The owner controls what information is captured, processed, shared, exported, deleted, or delegated to agents except where legal retention or explicit service policy applies.
Export right The owner must be able to export a reconstructable Core, not only download loose files. The export should include originals, events, knowledge, system definitions, selected derived data, manifests, rebuild instructions, and Core Bundles or Cotton Bundles for meaningful subsets.
Deletion right Deletion should remove or tombstone owner data according to clear policy while preserving auditability where required. Derived projections, indexes, caches, and backups need documented deletion or expiry behavior.
Takeover right The owner should be able to move from Core Managed to Core Self-Hosted or another compliant implementation without losing the event spine, semantic knowledge layer, glossary, taxonomy, bundles, and system definitions.
Boundary duty The service must prevent cross-owner leakage in storage, retrieval, embeddings, graph projections, logs, agent context, and operator tooling.
Transparency duty The service should make processing state, agent actions, data locations, retention rules, sharing state, and export/delete status inspectable by the owner.
Reversibility duty Specialized projections, indexes, and managed-service conveniences should remain rebuildable or replaceable unless explicitly promoted by governance.

Deletion governance must explicitly account for immutable originals and append-oriented events. If an object is under active retention, Core should write a tombstone or erasure-request event immediately, stop exposing the item through retrieval and derived views, delete or expire derived projections, and complete physical deletion when retention permits. Where per-owner or per-zone KMS boundaries are used, crypto-shredding through governed key deletion can be part of the erasure path, but it must be owner-ratified, audited, delayed according to cloud key-deletion rules, and paired with tombstone metadata so later systems can explain why retained ciphertext or manifests can no longer be read. Emergency governance bypass should remain narrowly scoped and logged.

Break-glass access must be tested as an operational path, not only specified in policy. For managed AWS, a role trust condition that requires MFA is not sufficient unless the named principal actually has MFA enrolled and a live sts:AssumeRole check with MFA succeeds. A negative assume-role test without MFA should remain denied, and the positive test should not perform any destructive or Governance-bypass operation. Owner-data migration should stay blocked until this emergency-access path is both constrained and usable.

Export and takeover rights should be operationalized through the Core Template, full Core Export Bundle, and Provisioning Manifest model in §21.1. A fiduciary service should be able to produce a reconstructable full export bundle, identify the exact template or compatible implementation profile needed to run it, and provide the manifest values and rebuild instructions required for the owner or a successor operator to stand up the Core elsewhere. This does not require exposing service secrets, but it does require enough structure, provenance, checksums, and re-encryption guidance for a destination Core to be validated.

21.7 Owner Visibility and Escalation Governance

Owner-facing visibility surfaces are curated projections over Core. They help the owner understand current state, act on decisions, and review important changes, but they are not the canonical source of truth. Canonical sources, event records, knowledge objects, manifests, and system definitions should change first; visibility surfaces should reflect the owner-relevant consequence.

Level Meaning Expected behavior
Internal Canonical source changed but owner-visible state did not materially change. No visibility update or notification required.
Visible only A curated owner-facing view should change, but the change is not notable. Update the view; no proactive notification.
Notable The change affects architecture, workflow, reliability, governance, active work, or something the owner should see soon. Create a concise owner-facing update or log entry with provenance.
Important The change is time-relevant, significant, blocked, failed, recovered, or likely to need owner attention. Notify according to owner preferences and deep-link to the exact relevant item where possible.
Approval The proposed action is sensitive, destructive, costly, publishing-related, externally visible, irreversible, or otherwise guarded. Use a request-scoped approval or verification path. Notification alone is not permission.

Notification preferences should be owner-controlled and category-aware. Useful generic categories include agent attention, agent approval, important visibility updates, operations failure, operations recovery, and daily or periodic summaries. Delivery state should distinguish at least delivered, seen/opened, verified where required, resolved, and intentionally skipped.

Deep links should prefer the most specific owner-facing item available: a completed-work detail page, update card, approval request, service incident, source analysis, Work Starter, or exact History entry. Generic landing pages are acceptable only when no better target exists.

21.8 System Inspection Surface

Core should have a read-first System surface for inspection, explainability, and governance. This surface is not the everyday capture surface and should not become the default place where the owner edits canonical content. Its purpose is to answer what happened, why it happened, what state changed, which agent or workflow acted, and what evidence or rule justified it.

Inspector area Purpose
Context inspector Shows retrieval logs, context packets, active filters, selected memory, token budgets, and why semantic/vector search was or was not used.
Rules and proposals Shows proposal history, rejection signal, confirmed rules, rule-fire logs, review due dates, and confidence fallbacks.
Runs and jobs Shows workflow runs, agent tasks, processing jobs, source imports, retries, failures, partial completions, and produced outputs.

The System surface should render existing Core records and logs rather than computing truth client-side. Powerful controls can be added gradually, but early versions should be read-first to avoid accidental truth changes.

21.9 Proposals, Rules, and Earned Autonomy

Core should treat AI autonomy as earned and scoped. An agent may observe, detect, draft, classify, and suggest, but structured suggestions that affect durable knowledge, profile truth, sharing, cost, automation, or governance should be recorded as proposals before they become canonical changes unless a standing rule already covers the exact class of action.

Object Purpose Governance rule
Proposal A structured suggestion with rationale, evidence, confidence, expected result, and owner response state. Rejected, edited, expired, and superseded proposals remain signal for future behavior.
Rule A confirmed standing permission for a defined class of future action. Created only through owner confirmation or explicit governance migration.
Rule fire log An audit record that a rule acted autonomously. Every autonomous rule execution should be inspectable, reversible where possible, and linked to its resulting object.
Confidence fallback Behavior when a rule match is weak, ambiguous, sensitive, costly, or unusual. Fall back to a proposal or approval request instead of acting silently.

Rejections are not failures. They teach the system what not to repeat. Confirmations can reduce future burden, but only inside the scope the owner approved.

21.10 Governance of Agent Definitions

Agent definitions, skills, roles, and operational files are part of Core. Runtime copies are not the primary truth. The Coordinator Agent manages the sync from Core to places where agents actually run.

This makes the system self-describing: Core knows what agents exist, what roles they play, which skills they use, where they run, and which copies are current.

21.11 Code, Runtime, and Knowledge Location Split

Core should separate durable information from runnable software and machine-local runtime state. This keeps the personal information architecture portable while allowing implementations to use normal software engineering practices.

23. Implementation Roadmap

23.1 Cotton v1 Foundation Gate

  • Use this master document as the canonical architecture reference.
  • Formalize Cotton v1 as the Context Blueprint used by Core Managed AWS, Core Self-Hosted, and any future implementation.
  • Create initial Cotton principles, taxonomy families, glossary-seeding rules, reference templates, Personal Operating Context templates, Work Starter templates, and agent instructions.
  • Formalize the Track 1/Track 2 adoption pattern, core_disposition axis, event-type registry, owner-profile subtypes, and ingestion gate governance as part of the first implementation-ready Cotton/Core governance package.
  • Use the completed historical mining pass as the first empirical grounding for information-type families, glossary-seeding rules, source-bias caveats, and domain coverage gaps.
  • Validate Cotton against representative non-work and non-architecture information types using a mix of available owner sources, public examples, synthetic profiles, mock records, and common export schemas. The goal is framework coverage, not complete personal-data mining.
  • Migrate a representative slice of existing PIOS 1.0 / personal vault information into the PIOS 2.0 Cotton model.
  • Use that migration to validate taxonomy values, context dimensions, glossary seeding, reference templates, storage recommendations, and agent behavior before locking implementation details.

23.2 Immediate Priority - Core Managed AWS

  • Start implementation from the AWS-native Core Managed path.
  • Create one owner-isolated S3 bucket with prefixes for the five Core data zones, then implement the minimal event envelope.
  • Build minimal inbox ingestion: upload, preserve original, create event, record source reference, and preserve source-original timestamps separately from Core primary-record timestamps.
  • Implement the source promotion lifecycle: parked, analyzed, mapped, enriched, and History-included.
  • Build the full Core Export Bundle and Provisioning Manifest tooling needed for rebuild, transfer, and takeover: export manifest, checksums, import/hydration, destination KMS re-encryption, parity validation, and clear authorization gates.
  • Measure real costs and processing bottlenecks before adding optional services.

23.3 General Framework Extraction

  • Keep PIOS framework concepts independent from AWS service names.
  • Extract stable specs for Cotton principles, taxonomy values, events, glossary terms, context metadata, Personal Operating Context, Work Starters, source definitions, retrieval contracts, export format, and agent definition format.
  • Keep existing-system transition specs implementation-neutral: functional-area mapping, disposition, bounded batches, peer/checklist/owner gates, parity proof, and decommission-after-parity should work in Core Managed and Core Self-Hosted.
  • Use AWS implementation learnings to improve the general framework without making AWS a dependency.

23.4 Core Self-Hosted Follow-Up

  • Maintain the Core Self-Hosted implementation contract while AWS remains the reference build, so AWS-specific services do not become accidental PIOS requirements.
  • Package the same information model on Ubuntu, Garage, Python, DuckDB, WireGuard, and Restic.
  • Define the self-hosted Core Template as a versioned data-empty image-root, VM image, install script, container composition, or configuration bundle that can create an empty Core Self-Hosted structure before bundle hydration and pass a hygiene gate before distribution.
  • Define self-hosted equivalents for large-file ingest, append-only or immutable originals/events, remediation tombstones, erasure/key custody, and restore proofs before broad migrated owner data is hydrated.
  • Treat local bundle hydration as the first reconstruction proof; the next self-hosted milestones are a booted VM proof, protected-ingestion parity, hardened key provider, repeatability across a second VM, and only then full self-hosted product hardening.
  • Document operational responsibilities clearly: updates, backups, keys, monitoring, and recovery.

23.5 Agent System and Circle Integration

  • Define Main, Coordinator, Orchestrator, Organizer / Inbox, Enrichment, Knowledge Maintenance, Pattern, Retrieval, Reflector, Time Planner, Responsibility, and Worker agent definitions in Core. Information Architect may remain a grouping label for the Organizer, Enrichment, Knowledge Maintenance, Pattern, and Retrieval agents rather than a separate required agent.
  • Implement Coordinator sync from Core to runtime AGENTS.md/skill files.
  • Treat runtime instruction files, skills, memory files, and MCP/tool configuration as governed system artifacts with canonical source copies and deployment sync state.
  • Give the Time Planner Agent access to the Dynamic Scheduling Model, Personal Operating Context, goals, targets, projects, Work Starters, tasks, calendar context, available windows, and completion history.
  • Give agents an update-first close-out rule for meaningful completed work so owner-facing History is generated from structured updates and manifests instead of reconstructed primarily from raw traces.
  • Give agents a plan-first validation rule for complex cross-system, source-ingestion, security, owner-facing, or data-integrity work.
  • Implement owner visibility governance so changes are classified as internal, visible only, notable, important, or approval before notification or escalation.
  • Expose agent PMs and group collaboration through Circle.
  • Ensure Circle conversations, decisions, outputs, and tasks flow back into Core.

Governing implementation rule: begin with durable storage, minimal event truth, and simple retrieval. Add specialized systems only when actual captured data, real retrieval patterns, and demonstrated performance needs explain precisely why they are required.
